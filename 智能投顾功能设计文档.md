## 1. 模块定位与子页面结构

### 1.1 模块定位

智能投顾作为第 6 大功能模块，与「股票查询」「批量打分」「基金分析」「模拟分析与迭代」「智能问答」并列在侧边栏。侧边栏展开子页面导航。

### 1.2 子页面结构

```
智能投顾
  ├── AI顾问对话    （核心入口 — 常驻右侧面板）
  ├── 股票推荐      （推荐结果展示）
  ├── 我的持仓      （持仓组合管理）
  ├── 策略市场      （策略选择/配置）
  ├── 回测&模拟盘   （运行与结果）
  └── 收益报告      （对比分析与 PDF 导出）
```

### 1.3 页面布局

每个子页面采用「左侧内容 + 右侧 AI 顾问常驻面板」布局：

```
┌──────────────────────────────────┬───────────────────┐
│ 左侧：子页面内容                  │  右侧：AI 顾问面板   │
│ （推荐结果 / 持仓表格 / 策略列表  │  常驻对话窗口       │
│  回测图表 / 报告预览）            │  上下文感知当前页面  │
│                                  │  知道用户在看什么    │
│                                  │  可就页面内容直接提问 │
└──────────────────────────────────┴───────────────────┘
```

用户在「股票推荐」「我的持仓」「策略市场」等任何子页面操作时，右侧 AI 顾问始终可见，且**感知当前页面上下文**。例如用户在「我的持仓」页面看到组合，直接在右侧问"我这个持仓里银行股占比是不是太高了？"——AI 不需要用户重复上下文。

## 2. AI 顾问对话引擎

### 2.1 模型配置

| 配置项 | 值 |
|--------|-----|
| 模型 | MiMo-V2.5-Pro（M1，当前本地最强模型） |
| 思考模式 | ✅ 开（最大思考深度） |
| 防幻觉机制 | 强化：数据事实区 / 分析判断区两区分离 + 来源标注 |

### 2.2 与智能问答的对比

| | 智能问答（现状） | 智能投顾（设计） |
|---|---|---|
| **主模型** | M5 MiMo-V2.5 | M1 MiMo-V2.5-Pro |
| **Pro 升级** | M1 MiMo-V2.5-Pro（复杂问题触发） | 直接用最优模型，不走分级 |
| **L1 快速路径** | 12s 超时 + 精简工具集 | 复用，但工具集需扩展 |
| **用户画像** | 无 | ✅ 注入 system prompt |
| **MCP 工具集** | 标准（行情/财务/资金流/板块等） | 标准 + 持仓操作 + 策略配置 + 回测控制 |
| **股票推荐** | ✅ 共享后端链接 | ✅ 内置 + 可加入持仓 |
| **持仓/策略/回测操作** | ❌ | ✅ 独占 |
| **页面上下文感知** | 无 | ✅ 知道用户当前页面和数据 |

### 2.3 多模态需求判断

**不需要多模态**。AI 面板的"上下文感知"通过程序化方式实现：前端将当前页面的结构化数据（表格、列表、参数、图表数据）在每次对话时作为上下文注入 system prompt。不需要 AI 真正"看"屏幕截图。

除非未来要支持"用户上传 K 线截图让 AI 分析形态"，才需要视觉多模态能力。当前版本不涉及。

### 2.4 System Prompt 组成

```
[系统角色] 金融投顾 AI 助手
[当前页面上下文] 用户在"我的持仓"页面，持仓数据: {...}
[用户画像] {user_profile_summary}
[可用工具] 标准 MCP 工具 + 投顾专属工具（持仓/策略/回测）
[防幻觉规则] 两区分离 + 来源标注 + 不确定时明说
[推荐免责] 所有推荐结果附："基于 AI 初筛，深入研究请使用打分及报告功能"
```

## 3. 用户画像系统

### 3.1 技术路线

采用 **C 方案：独立 JSON 持久化 + LangGraph 状态注入**。

```
对话 → LLM 调用 update_profile 工具 → UserProfileManager
  ↑                                              ↓
State 注入 ← start_node 加载 ←────────── user_profile.json
```

### 3.2 存储层

- **路径**：`data/user_profiles/{session_id}.json`
- **格式**：

```json
{
  "risk_tolerance": "Balanced",
  "investment_horizon": "Long-term",
  "favorite_sectors": ["科技", "消费"],
  "avoid_sectors": ["房地产"],
  "investment_style": "偏好高ROE、稳定分红的行业龙头",
  "custom_preferences": {
    "max_single_position_pct": 30,
    "prefer_etf": true
  },
  "updated_at": "2026-06-28T15:30:00"
}
```

**字段说明**：

| 字段 | 类型 | 枚举/说明 |
|------|------|-----------|
| `risk_tolerance` | string | `Conservative` / `Balanced` / `Aggressive` / `Unknown` |
| `investment_horizon` | string | `Short-term` / `Medium-term` / `Long-term` / `Unknown` |
| `favorite_sectors` | array | 偏好板块列表 |
| `avoid_sectors` | array | 回避板块列表 |
| `investment_style` | string | 自由文本描述 |
| `custom_preferences` | object | 任意扩展 key-value |

### 3.3 注入层

AI 顾问对话启动时：
1. 从 `data/user_profiles/{session_id}.json` 加载画像
2. 写入 `AgentState.data["user_profile"]`
3. 拼接进 system prompt 的「用户画像」段落
4. 每次收到新消息时 state 自动携带，LangGraph 的 `merge_dicts` reducer 确保并行安全

### 3.4 更新层

注册两个 MCP 工具供 LLM 调用：

**`update_user_profile`**：
- 参数：`risk_tolerance`, `investment_horizon`, `favorite_sectors`, `avoid_sectors`, `investment_style`, `**kwargs`
- 枚举约束防止 LLM 写入无效值
- 描述中写明："当用户明确陈述偏好或你可以从对话中推断时调用"
- `kwargs` 全部收入 `custom_preferences`，允许无限扩展

**`get_user_profile`**：
- 无参数，返回完整画像 JSON
- 用于 LLM 主动查询当前画像

### 3.5 LLM 自主决策

LLM 自己判断何时调用 `update_user_profile`，无硬编码规则：

| 用户说 | LLM 行为 |
|--------|---------|
| "我是保守型投资者" | 调用 `update_user_profile(risk_tolerance="Conservative")` |
| "我只喜欢分红稳定的股票" | 调用 `update_user_profile(investment_style="偏好高分红股票")` |
| "推荐一只股票" | 读取 profile 发现 Conservative → "基于您偏好低风险..." |
| （几天后）"推荐股票" | profile 跨会话持久化，记忆不丢失 |

## 4. 股票推荐链路

### 4.1 数据来源

推荐依托「模拟分析与迭代」功能产出的精筛股票池（`stock_pool.json` 中的 short/medium/long 三个期限池去重），由 LLM 根据用户需求从中挑选。全程**不触发任何分析 Agent**。

### 4.2 缓存使用规则

| 缓存类型 | 路径特征 | 推荐链路中的权限 |
|---------|---------|----------------|
| **生产打分缓存** | `intermediate_cache/` 的 `short/medium/long_term_score` | ✅ 直接用于排名 |
| **7 Agent signal_pack 缓存** | `intermediate_cache/` 的 `{agent}_signal_pack` | ✅ 可读，地位等同 PE/PB 等外部数据，仅参考 |
| **Agent 分析文本** | `{agent}_analysis` | ❌ 不调用 |
| **Eval/模拟盘缓存** | `eval/cache/` 的 `_eval` 后缀 | ❌ 绝对不可碰 |
| **MiMo-V2.5 模拟盘打分** | 任何在 Page 06 模拟分析与迭代中产生的 M5 数据 | ❌ 不具参考意义，仅适用于 Page 06 |

### 4.3 推荐算法（n/x/5 规则）

```
用户输入需求
    ↓
LLM 从精筛池（short/medium/long 三池去重）按需求语义初筛
    ↓
n = 满足用户需求的股票数
    ↓
n ≤ 5 ──→ 直接推荐 n 只 ✅
    ↓
n > 5 ──→ 查 n 只中哪些有生产打分缓存（intermediate_cache）
    ↓
x = 有打分缓存的股票数
    ↓
x = 0 ──→ 强力模型对 n 只做快速基本面分析
          · 仅调用基础数据 MCP 工具：daily_basic（PE/PB）、daily（走势）、stock_basic（行业/市值）
          · 可参考 signal_pack 缓存（若有），地位等同于外部数据
          · 不触发任何分析 Agent
          · 挑最好的 5 只 ✅
    ↓
x ≤ 5 ──→ 推荐这 x 只 ✅
    ↓
x > 5 ──→ 推荐 x 只中打分最高的 5 只 ✅
```

### 4.4 关键约束

- 精筛池保入口质量，不从头全市场筛选
- 打分缓存定优先级（生产模型 M1/M3 在 `intermediate_cache/` 下的正式打分）
- 无缓存时用强力模型轻量挑选，保证响应速度
- 全程不触发任何分析 Agent（不调 `run_stock_analysis` 等）
- 推荐结果须附免责声明："以上仅为 AI 初筛推荐，若需深入研究请使用「股票查询」中的打分及报告功能"

### 4.5 智能问答复用

智能问答页面的推荐功能走后端同一条链路（`src/advisory/recommendation.py`），区别仅在前端：智能问答不展示"加入持仓"按钮，智能投顾展示。

## 5. 自建持仓池与组合评估

### 5.1 持仓池管理

#### 5.1.1 添加来源

1. **从推荐添加** — 推荐页面每只股票旁有"加入持仓"按钮
2. **手动搜索添加** — 通过搜索框直接输入股票代码/名称

#### 5.1.2 权重逻辑

权重是**持仓市值的实时占比**，不是用户设定的目标比例：

```
权重_i = (持股数量_i × 当前价格_i) / 总持仓市值 × 100%
```

用户买入多少股、什么价格，权重自然确定。策略调仓和手动买卖后，权重实时重算，与实盘逻辑一致。

#### 5.1.3 前端展示

```
我的持仓
┌──────────────────────────────────────────────────────────────────┐
│ 股票代码    名称     持仓数量   成本价   现价   市值     权重    │
│ 600519.SH  贵州茅台   200股    1650    1680  336,000   31.8%    │
│ 000858.SZ  五粮液    1000股    148     152   152,000   14.4%    │
│ 600036.SH  招商银行  2000股     36      38    76,000    7.2%    │
│ 002415.SZ  海康威视  2500股     32      36    90,000    8.5%    │
│ ...                                                              │
│ ─────────────────────────────────────────────────────────────── │
│ 总市值: 1,056,500  总成本: 1,020,000  浮动盈亏: +36,500 (+3.6%) │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 组合健康度（五色分类）

#### 5.2.1 定位

面向整个持仓组合的综合评估，不针对单只股票，不区分长中短线。

LLM 综合判断，内置专业提示词。**这是一个综合过程，只给出大致的持仓评级，不给出具体分数。**

#### 5.2.2 颜色分级

| 颜色 | 含义 | 触发条件（LLM 综合判断） |
|------|------|------------------------|
| 🟢 深绿 | 组合稳健优秀 | 结构合理、风险分散、个股质量高 |
| 🟢 浅绿 | 组合良好 | 整体健康，少数个股或配比可优化 |
| 🟡 黄色 | 组合一般 | 存在明显短板（如行业过度集中、单股权重过高） |
| 🟠 橙色 | 需要关注 | 多处风险信号，建议调整 |
| 🔴 红色 | 存在显著风险 | 高风险集中、财务暴雷迹象、市场环境极端不利 |

#### 5.2.3 判断依据优先级

```
1. 有生效打分缓存（intermediate_cache）→ 重点参考
2. 有 signal_pack 缓存 → 作为参考
3. 调 MCP 基础数据（PE/PB/经营数据/走势）→ 自行判断
4. 打分缓存越全 → 越依赖分数，减少 LLM 自行判断成分
```

#### 5.2.4 输出格式

颜色评级 + 简短文字说明（如"医药行业占比过高且两只医药股权重合计达 65%"）。不输出具体分数。

#### 5.2.5 AI 投资建议

组合健康度下方附带 AI 建议区，内容：

- 整个持仓偏向什么期限投资（短期/中期/长期）
- 持仓中不同股票分别有什么建议（不针对单只股票打分，而是文字建议）

### 5.3 偏好匹配度（独立显示）

#### 5.3.1 定位

与组合健康度并列，回答"这个持仓适合你吗"的问题。

一个好的持仓不一定匹配用户需求，一个匹配用户需求的持仓不一定收益好。两个维度独立展示。

#### 5.3.2 判断方式

LLM 轻量调用工具，对比持仓股票基本信息（行业/市值/市盈率/分红率） vs 用户画像（风险偏好/偏好板块/回避板块/投资期限）。

#### 5.3.3 输出等级

| 等级 | 含义 | 说明 |
|------|------|------|
| ✅ 高度匹配 | 持仓与用户画像高度一致 | 行业/风险/期限都契合 |
| 👍 基本匹配 | 大体一致，个别偏离 | 如偏好科技但持仓中科技占比略低 |
| ⚠️ 部分偏离 | 存在明显不匹配项 | 如保守型用户持有高波动股票 |
| ❌ 显著偏离 | 持仓与用户画像严重矛盾 | 如回避板块大量持有、风险等级完全错位 |

附一句话原因。不消耗 Agent，轻量完成。

## 6. 策略系统

### 6.1 架构总览

采用 **fin-agent 风格策略类架构**：策略即 Python 类，`@register_strategy` 装饰器注册，纯代码执行，无 LLM 参与。

### 6.2 策略基类

```python
class TradingStrategy(ABC):
    name: str                          # 唯一标识，如 "ma_cross"
    description: str                   # docstring — LLM 可读的策略说明

    def enrich(self, df: pd.DataFrame, params: dict) -> pd.DataFrame:
        """在日线数据上添加指标列并返回"""

    def signal(self, row, prev_row, params, context) -> int:
        """返回 1(买入) / -1(卖出) / 0(不动)"""

    def risk_exit(self, row, prev_row, params, context) -> int:
        """持仓时优先判断止损/止盈，默认无。返回 -1 强制平仓，否则 0"""

    def position_fraction(self, params, context) -> float:
        """仓位比例，默认满仓 1.0"""
```

### 6.3 策略注册器

`StrategyRegistry` 单例模式，策略类通过 `@register_strategy` 装饰器自动注册：

```python
@register_strategy
class MACrossStrategy(TradingStrategy):
    name = "ma_cross"
    description = "双均线金叉买入、死叉卖出。参数：short_window(5), long_window(20)"
    # ... enrich, signal 实现
```

### 6.4 内置策略清单

从 fin-agent 移植并适配 A 股市场，共计 20+ 种：

| 分类 | 策略名称 | 说明 |
|------|---------|------|
| 均线 | `ma_cross` | 双均线金叉买/死叉卖 |
| 均线 | `triple_ma` | 三均线排列与短穿中 |
| 均线 | `ema_sma_bias` | 价穿 EMA 且 EMA>SMA 乖离 |
| 指标 | `macd` | DIF/DEA 金叉死叉 |
| 指标 | `rsi` | RSI 穿越超卖/超买阈值 |
| 指标 | `kdj` | K/D 金叉死叉 |
| 布林 | `boll_reversion` | 下轨外收回买/上穿中轨卖 |
| 布林 | `boll_breakout` | 突破上轨买/跌回中轨下卖 |
| 通道 | `donchian_breakout` | 唐奇安：破N日高买/破N日低卖 |
| 通道 | `turtle` | 完整海龟交易系统（含 ATR 止损） |
| 震荡 | `cci` | CCI 穿越超卖/超买线 |

| 复合 | `rsi_ma200` | RSI + 长期均线过滤 |

| 动量 | `momentum_roc` | N 日 ROC 过零交易 |

### 6.5 用户使用路径

#### 路径 1：拖拽选择

策略市场页面展示所有内置策略，每张卡片显示策略名称、一句话描述、适用场景。用户拖拽策略卡片到持仓组合上即完成绑定。

#### 路径 2：自然语言（右侧 AI 顾问）

用户说"我想对持仓里的科技股设一个 10% 止损线"：

1. AI 检索 `StrategyRegistry` 找到最匹配的已有策略
2. 如果已有策略不完全匹配 → LLM 复制最接近的策略文件 → 修改参数/逻辑 → 生成新策略 `.py` 文件 → `@register_strategy` 注册
3. 绑定到该持仓组合

前端提示："想要配置更复杂的策略？在右侧 AI 顾问中详细描述你的需求，AI 会为你定制策略。"自然语言支持比拖拽更复杂的策略配置。

#### 自定义策略文件位置

- 内置策略：`src/advisory/strategies/builtin/`
- 用户定制策略：`data/strategies/custom/{strategy_name}.py`
- 内置策略不可修改，用户定制策略可随时调整

### 6.6 LLM 在策略系统中的角色

**LLM 只在策略代码生成阶段参与，不参与策略执行。**

```
用户自然语言描述策略
    ↓
LLM：理解需求 → 检索已有策略 → 生成/修改策略 Python 文件
    ↓
策略文件 (.py) 写入磁盘 + 注册
    ↓
LLM 退出
    ↓
策略执行 = 纯 Python 代码：日线数据 → signal() → 1/-1/0
```

### 6.7 策略执行模式

| 模式 | 触发条件 | 调仓执行 |
|------|---------|---------|
| **全自动** | 持仓绑定了策略 | 收盘后自动计算信号 → 次日开盘自动执行调仓 |
| **全手动** | 持仓未绑定策略 | 用户手动在持仓页面买卖，无策略建议 |

没有"半自动待确认"模式。逻辑二分，干净利落。

## 7. 回测与模拟盘

### 7.1 统一引擎架构

回测和模拟盘共用同一个**策略执行引擎**和**交易模拟器**，区别仅在于数据来源和时间方向：

```
                    ┌─ 策略执行引擎 ─┐
                    │  signal()       │
                    │  risk_exit()    │
                    │  position_frac()│
                    └────────┬────────┘
                             │  调仓指令 (1/-1/0)
                    ┌────────▼────────┐
                    │  交易模拟器      │
                    │  （复用 eval/    │
                    │   market_        │
                    │   simulator.py） │
                    └────────┬────────┘
                             │
              ┌──────────────┴──────────────┐
              │                             │
         ┌────▼────┐                  ┌─────▼─────┐
         │  回测    │                  │  模拟盘    │
         │ 历史日线  │                  │ 今日+未来  │
         │ 一次性跑完 │                  │ 每日结算   │
         └─────────┘                  └───────────┘
```

**交易模拟器约束**（复用 `eval/market_simulator.py`）：
- T+1 交割（当日买次日才能卖）
- 佣金 + 印花税
- 滑点假设
- 100 股一手整数倍成交
- 涨跌停无法成交
- 停牌无法交易
- 成交量约束

### 7.2 调仓时序（模型 3）

采用业界标准做法：**收盘信号 → 次日开盘执行**。

```
T 日 15:00 后 → 获取 T 日收盘数据 → 策略计算 signal() → 生成调仓指令
T+1 日 9:25  → 以 T+1 日开盘价执行调仓 → 更新持仓 → 记录 T+1 日结算
```

**选择理由**：
- 无前视偏差（不会用当天收盘价的信息在当天收盘时交易）
- 学术正确，符合真实交易习惯
- 与券商真实交易逻辑一致：今天收盘后分析决策，明天集合竞价执行
- Tushare 每日线数据包含 `open` 字段，完全支持
- 所有内置策略均为日线级别，适配此模型

### 7.3 回测模式

```
用户设定：持仓组合 + 策略（可选）+ 起止日期
    ↓
拉取历史日线（含 T-N 向前预热数据，确保指标计算正确）
    ↓
逐日回放：
  T 日收盘数据 → 计算 signal() → 生成调仓指令
  T+1 日以开盘价执行 → 更新持仓 → 记录日结算
    ↓
输出：
  · 总收益率
  · 年化收益率
  · 最大回撤
  · 夏普比率
  · 交易次数
  · 权益曲线数据
  · 完整交易记录列表
```

### 7.4 模拟盘模式

```
用户设定：持仓组合 + 策略（可选）+ 初始资金
    ↓
从建仓日或当前日期开始运行
    ↓
每个交易日收盘后自动运行：
  有绑定策略 → 计算 signal() → 生成调仓指令
  无绑定策略 → 跳过（等待用户手动操作）
    ↓
次日开盘执行调仓 → 更新持仓 → 记录日结算
    ↓
持续运行直到用户手动停止
```

### 7.5 手动交易

用户在「我的持仓」页面点击某只股票 → 弹出交易面板：

```
┌─ 交易操作 ─────────────────────┐
│ 股票: 贵州茅台 (600519.SH)      │
│ 操作: [买入 ●] [卖出 ○]        │
│ 数量: [____] 股                 │
│ 价格: [按次日开盘价 ▼]          │
│       ○ 按次日开盘价（默认）     │
│       ○ 按指定价格              │
│       ○ 精确到分钟（高级）       │
│                                │
│ 预计成交: ~336,000 元           │
│ 当前可用资金: 520,000 元        │
│                    [确认下单]    │
└────────────────────────────────┘
```

- 默认以次日开盘价成交
- 用户可选指定价格（回测时以该价格判断是否可成交）
- 高级用户可选分钟级精确价格（需 Tushare `stk_mins` 数据支持）

### 7.6 分钟级交易（高级功能）

Tushare `stk_mins` 支持 A 股 1min/5min/15min/30min/60min 五种粒度的历史分钟数据。

| 模式 | 交易粒度 | 结算价格 | 适用场景 |
|------|---------|---------|---------|
| **策略自动** | 日级 | 次日开盘价 | 所有内置策略 |
| **手动-快速**（默认） | 日级 | 次日开盘价 / 用户指定日级价格 | 日常手动交易 |
| **手动-精确**（高级） | 分钟级 | 用户选时间，查 `stk_mins` 取该分钟 close | 复盘历史交易 |

分钟级仅为手动交易提供精细入账能力，不改变策略系统日级执行的本质。

### 7.7 自动追赶机制

#### 7.7.1 触发方式

用户打开项目 → 进入智能投顾板块 → **自动检测并触发**，无需用户手动操作。

#### 7.7.2 追赶逻辑

```
进入智能投顾板块
    ↓
检测上次结算日期 T_last（从结算记录中读取）
    ↓
计算 missed_days = 今天 - T_last 之间的交易日数
    ↓
missed_days = 0 → 正常状态，不追赶
missed_days > 0 → 自动追赶
    ↓
逐日回放：
  Day_1: 获取 Day_1 收盘数据 → 跑策略 signal() → Day_2 开盘价执行结算
  Day_2: 获取 Day_2 收盘数据 → 跑策略 signal() → Day_3 开盘价执行结算
  ...
  Day_N: 获取 Day_N 收盘数据 → 跑策略 signal() → 若今日未开盘则暂存待执行
    ↓
更新持仓、权益曲线、结算记录
    ↓
前端通知："已自动追赶 5 个交易日（06/24 → 06/28）"
```

#### 7.7.3 与 Eval 系统追赶机制的区别

| | Eval 系统 | 智能投顾模拟盘 |
|---|---|---|
| **单日内容** | 跑 7 个 Agent + 3 个 Scorer → LLM 密集型 | 拉日线 → 策略 signal() → 纯 Python |
| **单日耗时** | ~200 秒/股票 | <1 秒（整个持仓组合） |
| **追赶上限** | 最多 7 个交易日 | **不设上限**（30 天约 30 秒） |
| **触发方式** | 手动/自动 | 进入板块自动触发 |
| **LLM 开销** | 有（M5 打分/分析） | 零 |

#### 7.7.4 追赶保证

- 策略信号的产生与实际情况**完全一致**（同一日数据、同一策略逻辑）
- 追赶过程中策略信号与当日如果在线运行时产生的结果完全相同
- 全部纯 Python 代码执行，速度快，无 LLM 开销

### 7.8 回测与模拟盘的关系

用户可同时为同一持仓组合分别运行回测和模拟盘：

| | 回测 | 模拟盘 |
|---|---|---|
| **数据方向** | 历史 → | 现在 → 未来 |
| **策略** | 用户指定 | 用户指定（可与回测不同） |
| **起止** | 用户设定期限 | 从建仓日开始，持续运行 |
| **目的** | 验证策略历史表现 | 跟踪当前实际持仓 |
| **执行** | 一次性跑完 | 每日增量结算 |

## 8. DeepSeek 自由投资线对照

### 8.1 定位

用户的持仓 + 策略是一套组合拳，DeepSeek 自由线是**对照基准线**。同一初始资金、同一起跑日期，DeepSeek V4 Pro 完全自主决策，与用户持仓 PK 收益。

### 8.2 核心约束（总纲 §3.5 严格执行）

```
DeepSeek V4 Pro 可见：                     DeepSeek V4 Pro 不可见：
├── Tushare MCP 全部工具                   ├── 任何 Agent 的 analysis 文本
├── AKShare MCP 全部工具                   ├── 任何 signal_pack
├── 股票基础信息/行情/财务/资金流           ├── 任何 score（短期/中期/长期）
├── 宏观数据/板块/概念                     ├── 精筛池排名/股票推荐
├── 自己的历史交易记录和当前持仓            ├── 用户的持仓/策略/偏好
└── 可用资金状态                           ├── 用户画像
```

**关键原则**：只给原始 MCP 数据，不给任何本项目生产的中间产物。这跟总纲 §3.5 的 LLM Free 线约束完全一致。

### 8.3 运行流程

```
用户设定对比起点日期 + 初始资金（默认与用户持仓相同）
    ↓
每日收盘后，DeepSeek V4 Pro 收到：
  · 当日收盘数据（全部 A 股的 Tushare 日线）
  · 可用 MCP 工具列表
  · 当前自身持仓状态 + 可用现金
  · 指令："根据今日收盘数据，自主决定明日持仓调整"
    ↓
DeepSeek V4 Pro 输出调仓指令
    ↓
后端校验（资金是否足够 / 持仓是否合法 / 涨跌停约束 / T+1 约束）
    ↓
次日开盘执行 → 日结算
    ↓
和用户持仓同步运行，持续对比
```

### 8.4 前端展示

在「回测 & 模拟盘」页面，两条权益曲线叠加展示：

```
收益率对比图

 │  115% ┤                                      │
 │  110% ┤                    ╭─ 用户持仓（实线）│
 │  105% ┤               ╭───╯                  │
 │  100% ┤──────────────╯                        │
 │   95% ┤   ╭──╮                                │
 │   90% ┤──╯  ╰─────── DeepSeek 自由线（虚线）  │
 │       ┤                                       │
 │   06/01   06/10   06/17   06/24   06/30       │
 └─────────────────────────────────────────────┘

 用户持仓: +15.2% | 最大回撤 -8.1% | 夏普 1.35
 DeepSeek: +3.8%  | 最大回撤 -5.2% | 夏普 0.72
```

### 8.5 图表导出

用户可导出对比收益率曲线为 PNG 图片，用于个人记录或分享。

## 9. PDF 持仓收益分析报告

### 9.1 触发方式

在「收益报告」子页面，用户选择：

- **报告类型**：模拟盘报告 / 回测报告
- **时间范围**：起止日期
- **对照组**：是否对比 DeepSeek 自由线
- **是否包含具体交易明细**

点击"生成报告" → AI 撰写 → 预览 → 下载 PDF。

### 9.2 报告结构

```
1. 收益概览
   · 总收益率 / 年化收益率
   · 最大回撤 / 最大回撤持续天数
   · 夏普比率
   · 胜率（盈利交易占比）
   · 与基准对比（沪深300 同期表现 / DeepSeek 自由线表现）

2. 持仓变化时间线
   · 按时间线列出每次调仓：
     - 日期 / 动作（买入/卖出）/ 标的 / 数量 / 成交价 / 调仓原因
   · 策略触发信号标注（如"MACD 金叉触发买入"）

3. DeepSeek 自由线对比（若用户选择）
   · 两条权益曲线并排图
   · 持仓风格差异分析（行业偏好 / 换手率 / 集中度 / 平均持仓天数）
   · 关键分歧节点标注
     - 如："06/15 茅台大跌 -5%，DeepSeek 减仓 50% 而你选择加仓"
     - 标注每次分歧后的短期收益对比

4. AI 综合评价
   · 用户持仓亮点总结
   · 风险暴露分析（行业集中度、单股权重、系统性风险）
   · 策略有效性评估
   · 改进建议（具体可操作）

5. 附录
   · 完整交易日结算明细表
   · 策略参数配置记录
   · 数据源声明
```

### 9.3 技术实现

| 组件 | 实现方式 |
|------|---------|
| **收益计算** | 纯 Python（复用策略引擎 + 交易模拟器） |
| **图表生成** | `matplotlib` / `plotly` 生成静态图 |
| **AI 分析文本** | DeepSeek V4 Pro（M6），复用 eval `report_writer_agent.py` 模式 |
| **PDF 生成** | 复用现有 `weasyprint` / `fpdf` 链路 |
| **报告存储** | `data/reports/` 按日期归档，用户可随时回看 |

### 9.4 防幻觉

AI 撰写报告内容须标注数据来源：
- `[数据]` — 来自系统结算记录 / MCP 工具返回的确定数据
- `[判断]` — AI 的分析推断
- 数据部分不可编造，判断部分须有依据

# 项目结构蓝图

本项目为股票投资顾问 Agent 系统，包含后端分析引擎与前端交互界面。
本文档面向前端设计与后端协调的总体规划，用户可以看到两个页面：股票查询与股票池。

## 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 后端 API | FastAPI | 作为前后端桥梁，暴露 RESTful 接口 |
| 分析引擎 | LangGraph + 现有 Agent | 复用已有的 fundamental/technical/value/news/summary agent 及打分器 |
| 中间产物存储 | JSON 文件 + 滑动窗口 | 按股票+agent+日期命名，保留最近 7 天 |
| 前端 | Streamlit | `pages/` 多页面架构，纯 Python 实现 |
| LLM 调度 | 双模型策略 | 快速查询用低延迟模型，深度报告用强推理模型 |

## 页面一：股票查询

用户输入股票代码或股票名称，输入框右侧有两个按钮："查询"与"生成报告"。

### 查询按钮

点击后执行快速分析流程：

1. **模型选择**：调用低延迟、低幻觉率的模型（如 Haiku 级别），优先速度而非深度
2. **展示内容**：
   - 股票基本信息：市值、PB、PE、当日换手率
   - 涨跌幅度：近1日、近5日、近1月、近3月、近6月、近1年、近3年
   - 所属行业及简介、公司简介
   - **投资建议**：对该股票是否适合短线/中线/长线投资给出初步判断
     - 可以同时适合多种期限，也可以都不适合
     - 一切以事实为准，不做过度推荐

### 生成报告按钮

点击后执行深度分析流程：

1. **模型选择**：调用强推理模型（如 Opus/Sonnet 级别），确保报告质量
2. **执行流程**：通过 FastAPI 触发异步任务，依次调用 fundamental_agent、technical_agent、value_agent、news_agent
3. **中间产物**：四个 agent 的输出存入中间产物存储区（详见"中间产物管理机制"）
4. **报告生成**：将中间产物送入 summary_agent 生成详细分析报告
5. **报告下载**：
   - 报告标题格式：`股票名称_分析报告_YYYYMMDD.md`（如 `寒武纪_分析报告_20260426.md`）
   - 用户通过 `st.download_button` 下载到本地
   - 报告生成后保留 24 小时，过期自动清理（不依赖"下载完成"事件，因为该事件服务端无法感知）

## 页面二：股票池

分为短线投资池、中线投资池、长线投资池，三个 Tab 展示。股票由用户根据股票代码或名称自行添加。同一只股票可以同时在多个池中存在。

### 池内展示

每个池用表格渲染，每只股票占据独立一行：

```
股票名称 | 股票代码 | 当前打分 | 打分时间 | [打分] [生成报告]
─────────────────────────────────────────────────────────────────
寒武纪   | 688256  |  78.5   | 2026-04-26 14:30 | [打分] [生成报告]
比亚迪   | 002594  |  65.2   | 2026-04-25 09:15 | [打分] [生成报告]
```

- 分数和打分时间长期保留在前端页面，直到用户下次重新打分
- 打分时需标注打分的时间戳

### 打分功能（核心功能，需保证精确性）

不同期限的打分调用不同的 agent 组合和打分器：

| 期限 | 调用的 agent | 打分器 | 设计理由 |
|------|-------------|--------|----------|
| 短线 | technical_agent, news_agent | short_term_scorer | 短线主要受技术面和市场情绪驱动 |
| 中线 | fundamental_agent, technical_agent, value_agent, news_agent | medium_term_scorer | 中期需要综合考量 |
| 长线 | fundamental_agent, value_agent | long_term_scorer | 长线更看重基本面与估值，技术面和新闻噪音权重应降低 |

**打分逻辑**：
1. 先检查中间产物存储区：若对应时点已存在该 agent 输出，直接调用，避免重复消耗 token
2. 若不存在，运行对应 agent 生成中间产物并保留
3. 将相关 agent 输出送入对应打分器计算分数
4. 分数显示在股票行后方，同时记录打分时间戳

### 生成报告功能

与股票查询页面的生成报告功能完全一致，相当于同一功能提供了两个入口。

## 中间产物管理机制

### 存储规则

- 命名格式：`{agent_name}_{stock_code}_{YYYYMMDD}.json`
  - 示例：`value_agent_688256_20260427.json`
- 存储位置：`data/intermediate_cache/` 目录
- 格式：JSON 文件，包含 agent 名称、股票代码、日期、输出内容、生成时间戳

### 生命周期（滑动窗口）

1. **写入**：每次 agent 运行后，将输出写入对应日期的缓存文件
2. **读取**：调用 agent 前，先检查缓存中是否存在当日对应文件，存在则直接读取
3. **清理**：保留最近 7 天的中间产物，超过 7 天的文件自动删除
   - 保留 7 天而非仅 1 天的原因：允许用户对比历史分析变化，同时避免某些日期因 API 限流等原因导致数据丢失后无法回溯

### 中间产物与打分的联动

- 打分时优先从缓存获取 agent 输出
- 若缓存中无对应数据（从未跑过或已过期），则重新运行 agent，生成新的中间产物
- 一份中间产物可以被多次打分和报告生成复用

## 后端 API 设计（FastAPI）

### 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/query` | 快速查询（低延迟模型），返回股票基本信息+投资建议 |
| POST | `/api/report` | 触发深度报告生成（异步），返回任务 ID |
| GET | `/api/report/{task_id}` | 查询报告生成状态，完成后返回下载链接 |
| GET | `/api/pool/{term}` | 获取指定期限股票池内容（term: short/medium/long） |
| POST | `/api/pool/{term}` | 向指定期限股票池添加股票 |
| DELETE | `/api/pool/{term}/{stock_code}` | 从指定期限股票池删除股票 |
| POST | `/api/score/{term}/{stock_code}` | 对指定股票执行指定限期打分 |
| GET | `/api/cache/{stock_code}` | 查看某只股票的中间产物缓存状态 |

### 动态工作流编排

- 不再使用固定的 LangGraph 工作流
- 根据 API 请求类型动态选择要运行的 agent 子集
- 每次 agent 执行前先检查中间产物缓存

## Streamlit 前端设计

### 目录结构

```
app/
├── Home.py              # 入口，重定向到查询页
├── pages/
│   ├── 01_股票查询.py    # 股票查询页面
│   └── 02_股票池.py      # 股票池页面（三个 Tab）
├── components/
│   ├── query_form.py     # 查询表单组件
│   ├── result_card.py    # 查询结果卡片组件
│   ├── pool_table.py     # 股票池表格组件
│   └── score_display.py  # 分数展示组件
├── api_client.py         # FastAPI 客户端封装
└── config.py             # 前端配置（API 地址、模型选择等）
```

### 页面交互流程

**股票查询页**：
1. 用户在 `st.text_input` 输入股票代码/名称
2. 点击"查询"→ 调用 `/api/query` → `st.spinner` 显示加载 → 渲染结果卡片
3. 点击"生成报告"→ 调用 `/api/report` → 后台异步执行 → 完成后显示 `st.download_button`

**股票池页**：
1. Tab 切换（短线/中线/长线）
2. 用户通过输入框添加股票到当前池
3. 每行显示打分和生成报告按钮
4. 打分按钮触发后，显示 `st.spinner`，完成后更新该行分数和时间戳
5. 分数通过 `st.session_state` 保持状态

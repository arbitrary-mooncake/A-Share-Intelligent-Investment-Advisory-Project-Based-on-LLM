## 3. 架构总览

### 2.1 数据流（优化后目标状态）

```
用户问题
  │
  ├─ Step 1: 复杂度分析 (complexity_analyzer.py)
  │   └─ L0-L5 分级 + thinking_budget
  │
  ├─ Step 2: 股票/主题识别 (task_planner.py)
  │   └─ 股票代码提取 + 主题ETF匹配
  │
  ├─ Step 3: 任务规划 (task_planner.py)
  │   └─ 数据域 → 工具列表 (L4/L5也规划全量工具)
  │
  ├─ Step 4: 证据装配 (evidence_assembler.py) ← 本轮优化重点
  │   ├─ Phase 1: 全量并行预取 (所有级别)
  │   │   ├─ asyncio.gather 全量工具
  │   │   ├─ retry_failed_fetches 3轮重试
  │   │   ├─ alt_kwargs 备选参数回退
  │   │   ├─ 工具级内存缓存
  │   │   └─ 增强空结果检测
  │   ├─ Phase 1.5: 数据质量评估 (新增)
  │   │   ├─ 成功率统计
  │   │   ├─ 时效性检查
  │   │   └─ 冲突检测
  │   └─ Phase 2 (仅L5): ReAct补充钻探
  │       └─ 针对明确数据缺口做 ≤5步探索
  │
  ├─ Step 5: 运行时升级 (complexity_analyzer.py)
  │   └─ 基于实际数据质量调整level
  │
  └─ Step 6: 流式回答生成 (answer_generator.py) ← 本轮优化重点
      ├─ 两区强制输出 (📊数据事实区 / 🔍分析判断区)
      ├─ 数据质量上下文注入
      ├─ 时效性警告标注
      └─ 按level分级的max_tokens/thinking_budget
```

### 2.2 关键设计原则

1. **全量预取优于逐步探索**：A股QA问题边界明确，应一次性拉取所有相关数据，让LLM获得完整上下文
2. **数据清洗在LLM之前**：空结果过滤、格式标准化、冲突标注必须在注入prompt前完成
3. **证据与判断物理隔离**：两区输出是所有模板的强制要求，不可省略
4. **降级优雅、失败透明**：数据缺失时明确告知而非猜测；工具故障时快速失败而非卡住

### 3.2 空结果检测增强

**问题**：当前 `evidence_assembler.py` 第74行仅用 `len(text) > 25` 判断是否为空结果。分析Agent的 `fetch_utils.py` 有更完善的检测逻辑。

**具体改动**：在 `evidence_assembler.py` 中新增/修改：

```python
# 新增：增强版空结果检测（移植自 fetch_utils.py）
_EMPTY_RESULT_MARKERS = [
    "数据不可用", "工具不可用", "工具调用异常",
    "返回过短", "调用失败", "暂无数据",
    "查询结果为空", "未找到相关数据",
    "Error:", "Exception:", "HTTPError",
]

def _is_empty_tool_result(text: str) -> bool:
    """判断工具返回是否为有效数据（移植自 fetch_utils.is_empty_result）"""
    if not text:
        return True
    t = str(text).strip()
    # 规则1: 长度过短（提高阈值到50字符，原25太宽松）
    if len(t) < 50:
        return True
    # 规则2: 包含空结果标记
    if any(marker in t for marker in _EMPTY_RESULT_MARKERS):
        return True
    # 规则3: 只包含空白/换行/JSON空对象
    if t in ("{}", "[]", "null", "None", ""):
        return True
    return False
```

同时在 `_call_tool_safe()` 中（约第74行）将 `len(text) > 25` 改为调用 `_is_empty_tool_result(text)`。

**验收标准**：
- [ ] `_is_empty_tool_result` 函数已添加并使用
- [ ] 测试：模拟返回 `"数据不可用"` → 被正确识别为空结果并触发重试
- [ ] 测试：模拟返回 `< 50字符的短数据` → 被正确识别为空结果

### 4.1 移植 retry_failed_fetches 模式

**问题**：QA当前的重试逻辑简陋（固定2次重试，1.5s/3s间隔）。分析Agent使用成熟的 `fetch_utils.retry_failed_fetches()`：3轮递减并发(8→4→2)、不可恢复错误跳过、明确标注空结果。

**实施方案**：将 `src/utils/fetch_utils.py` 的 `retry_failed_fetches` 直接复用于QA系统。该函数接口为：

```python
async def retry_failed_fetches(
    results: list,           # 第一轮结果
    tool_infos: list,        # [(tool, kwargs), ...] None=占位符(工具不可用)
    labels: list,            # 每个工具的标签
    call_fn: callable,       # 重试调用函数 _call_tool_safe
    agent_label: str = "",   # 日志标签
    max_extra_rounds: int = 3,
    alt_kwargs_list: list = None,  # 备选参数（第0轮重试用）
) -> list:
```

**具体改动**：改造 `assemble_evidence_fast()` 的数据获取流程：

```python
# 旧代码（当前）:

         for tool, kwargs in zip(tools, kwargs_list)]

# 新代码（改造后）:
```

### 4.2 工具级内存缓存

**问题**：QA只有磁盘缓存（7天TTL），每次问答都做磁盘IO。同一会话中多次问答会反复调相同API。分析Agent的 `tool_cache.py`（5分钟内存缓存）可以零成本复用。

**实施方案**：在 `_call_tool_safe()` 中增加内存缓存层。直接复用 `src/utils/tool_cache.py`。

**具体改动**：在 `evidence_assembler.py` 的 `_call_tool_safe()` 函数开头增加：

```python
async def _call_tool_safe(tool, kwargs: dict, timeout: float, label: str,
                         session_id: str = "", max_retries: int = 2) -> str:
    tool_name = getattr(tool, 'name', 'unknown')

    # 第0层：工具级内存缓存（5分钟TTL，进程内去重）
    from src.utils.tool_cache import get_cached_tool_result, set_cached_tool_result
    cached = await get_cached_tool_result(tool_name, kwargs)
    if cached:
        logger.debug(f"QA Evidence: [工具缓存命中] {tool_name}")
        return cached

    # 第1层：会话级磁盘缓存（7天TTL）
    if session_id:
        cached = get_cached_evidence(session_id, tool_name, kwargs)
        if cached:
            return cached

    # 第2层：全局跨会话磁盘缓存（7天TTL）
    cached = get_global_cached_evidence(tool_name, kwargs)
    if cached:
        return cached

    # ... 实际调用工具 ...

    # 成功获取后回写三层缓存
    await set_cached_tool_result(tool_name, kwargs, text)  # 内存
    if session_id:
        set_cached_evidence(session_id, tool_name, kwargs, text)  # 磁盘
    set_global_cached_evidence(tool_name, kwargs, text)  # 全局
```

## 6. P2：ReAct路径重构 ✅ 已实施

> **目标**：将L4/L5从"纯ReAct"改为"全量预取 + ReAct补充钻探"。
> **改动文件**：`evidence_assembler.py`、`qa_engine.py`
> **预计工作量**：6-8小时

### 5.1 设计理念

当前：
```
L4/L5 → ReAct (20步, 14个工具) → 回答
```

目标：
```
L3/L4/L5 → Phase 1: 全量预取（与L2相同，但工具更多）
         → Phase 2: 数据质量评估
         → Phase 3 (仅L5, 且存在明确缺口): ReAct补充钻探(≤5步)
         → 回答
```

### 5.2 为什么要改

| 维度 | 纯ReAct（当前） | 全量预取+ReAct补充（目标） |
|------|----------------|--------------------------|
| 数据完整性 | 取决于Agent探索路径 | 全量预取保证基础覆盖 |
| 延迟 | 不可预测（20步×LLM推理） | 可预测（预取固定+可选的5步ReAct） |
| 幻觉风险 | 高（片段数据+多轮推理） | 低（完整数据+单次分析） |
| 工具故障容忍 | 低（一步失败影响后续） | 高（预取阶段有重试） |
| 适用场景 | 开放探索 | 有明确数据域的A股分析 |

```python
# Phase 1: 全量数据预取（以 fundamental_agent 为例）
async def fundamental_agent_node(state: AgentState) -> AgentState:
    stock_code = state["data"].get("stock_code", "")
    company_name = state["data"].get("company_name", "")

    # 1. 加载工具白名单
    TOOL_NAMES = [
        "tushare_stock_info", "tushare_income", "tushare_balancesheet",
        "tushare_cashflow", "tushare_fina_indicator", "tushare_daily_basic",
        "tushare_dividend", "tushare_ev_ebitda", "tushare_adj_factor",
        "tushare_top10_holders", "tushare_st_status",
    ]
    all_tools = await get_mcp_tools(tool_filter=TOOL_NAMES)
    tool_map = {t.name: t for t in all_tools}

    # 2. 构建并发任务
    tasks, labels, tool_infos = [], [], []
    for tname in TOOL_NAMES:
        if tname in tool_map:
            kwargs = _build_kwargs(tname, stock_code)
            tasks.append(_call_tool_safe(tool_map[tname], kwargs, tname))
            labels.append(tname)
            tool_infos.append((tool_map[tname], kwargs))
        else:
            tool_infos.append(None)  # 占位符

    # 3. 第一轮执行
    results = await asyncio.gather(*tasks, return_exceptions=True)

    # 4. 3轮递减并发重试
    results = await retry_failed_fetches(
        results, tool_infos, labels, _call_tool_safe,
        agent_label="FundamentalAgent",
        alt_kwargs_list=alt_kwargs_list,  # 可选
    )

    # 5. 清理结果
    safe_results = []
    for r in results:
        if isinstance(r, Exception):
            safe_results.append(f"[工具调用异常: {str(r)[:200]}]")
        else:
            safe_results.append(str(r) if r and len(str(r)) > 50 else "[空返回]")

    # Phase 2: LLM深度分析
    data_text = "\n\n".join(
        f"### [{label}]\n{result}" for label, result in zip(labels, safe_results)
    )

    llm = ChatOpenAI(
        model=model_name, api_key=api_key, base_url=base_url,
        temperature=1.0, max_tokens=16000,
        extra_body=get_thinking_body(base_url, True),
    )

    prompt = SYSTEM_PROMPT + f"\n\n## 原始数据\n{data_text}\n\n请分析并输出<SIGNAL_PACK>"
    response = await llm.ainvoke([HumanMessage(content=prompt)])

    # 6. 提取结构化证据
    signal_pack = _extract_signal_pack(response.content)
    return {**state["data"], "fundamental_analysis": response.content,
            "fundamental_signal_pack": signal_pack}
```

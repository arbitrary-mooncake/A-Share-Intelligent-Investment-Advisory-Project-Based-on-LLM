# 智能投顾项目新增“基金专区”功能开发蓝皮书

## 一、项目目标

为现有智能投顾项目新增一个“基金专区”功能，与已有的以下四大功能并列：

- 股票查询
- 智能问答
- 股票池
- 批量打分

基金专区同样需要具备以下两类核心能力：

- 打分能力
- 报告产出能力

---

## 二、总体架构设计

整个基金功能共设置 **9 个 agent**，包括：

- **7 个分析 agent**
- **1 个基金报告生成 agent**
- **1 个基金打分 agent**

建议采用如下执行流程：

1. **前 7 个分析 agent 并行运行**
2. 7 个分析 agent 的中间产物进入 **中间汇总节点**
3. 汇总完成后，根据用户前端指令，选择进入：
   - **Agent 8：基金报告生成 agent**
   - **Agent 9：基金打分 agent**

---

## 三、Agent 设计总览

### Agent 1：基金产品文件解析 agent

这是基金场景的“基础 agent”，优先级最高。

#### 作用

解析并结构化提取以下信息：

- 基金类型
- 投资目标
- 投资范围
- 投资策略
- 业绩比较基准
- 风险收益特征
- 申购赎回规则
- 费用结构
- 是否有最短持有期
- 是否为 ETF / LOF / 场外 / QDII / FOF / 养老目标等

#### 为什么必须有

基金的很多关键属性，不是从行情里得出来的，而是从法律文件里定义出来的。基金产品资料概要明确覆盖投资目标与投资策略、业绩比较基准、风险收益特征，并提示投资者在作出决定前阅读完整招募说明书等销售文件。(csrc.gov.cn)

#### 输出建议

```json
{
  "fund_type": "active_equity",
  "benchmark": "沪深300指数收益率*80% + 中债综合指数收益率*20%",
  "risk_level": "R4",
  "min_holding_period_days": 365,
  "subscription_fee": {},
  "redemption_fee": {},
  "style_tags": ["成长", "偏消费", "高仓位"],
  "doc_confidence": 0.95
}
```

---

### Agent 2：业绩与风险评价 agent

这是基金打分里的核心 agent 之一。

#### 作用

做基金层面的历史绩效与风险分析，例如：

- 年化收益
- 最大回撤
- 波动率
- 夏普 / 索提诺
- Calmar
- 同类排名稳定性
- 相对基准超额收益
- 回撤控制能力
- 不同市场阶段表现

#### 注意事项

这里要特别强调：它只能评价历史表现，不应把过去表现直接当成未来预测。

证监会关于基金评价业务的规则明确指出，基金评价结果并不是对未来表现的预测，也不应视作投资建议；关于基金宣传推介的监管要求也强调，基金过往业绩并不预示其未来表现。(csrc.gov.cn)

#### 输出建议

```json
{
  "perf_score": 78,
  "risk_score": 64,
  "alpha_vs_benchmark": 0.032,
  "max_drawdown": -0.28,
  "style_regime_stability": 0.71,
  "confidence": 0.83
}
```

---

### Agent 3：持仓穿透 / 组合结构 agent

这是基金功能里最有壁垒的 agent 之一。

#### 作用

分析基金披露组合，提取：

- 股票 / 债券 / 现金 / 其他资产占比
- 行业暴露
- 风格暴露
- 集中度
- 前十大持仓质量
- 与基准的偏离
- 是否存在“风格漂移”
- 是否存在“名义上是 A，实际上像 B”的问题

#### 为什么重要

基金定期报告会披露投资组合等内容，基金产品资料概要和法律文件也会披露产品定位、投资策略与基准。结合这些信息，可以判断基金是否“言行一致”。

同时需要注意，持仓信息存在披露滞后，通常只能看到最近一期定期报告末的持仓，不代表实时仓位，因此这个 agent 必须输出 **freshness / stale penalty**。

#### 和现有股票系统的关系

这个 agent 可以调用原来的股票基本面 / 估值 / 风险识别工具，对重仓股做穿透聚合，但要加上滞后折扣。

---

### Agent 4：基金经理 / 团队 / 管理人 agent

这个 agent 非常重要，尤其对主动基金。

#### 作用

评价以下内容：

- 基金经理任职年限
- 在管产品风格一致性
- 历史回撤管理能力
- 团队稳定性
- 管理人平台能力
- 是否频繁更换基金经理
- 同类产品线是否拥挤

#### 说明

这部分很多数据未必完全来自法规条文本身，但从基金披露框架和产品资料概要、定期报告、管理人报告等信息里可以抽取相当多结构化信息，因此这个 agent 在工程上是必要且可落地的。

定期报告模板会包含管理人报告、投资组合报告等内容，为这类 agent 提供了基础数据来源。(csrc.gov.cn)

---

### Agent 5：基准一致性 / 风格漂移 agent

这是强烈建议单列的 agent。

#### 作用

专门判断：

- 基金当前表现是否与其业绩比较基准逻辑一致
- 基金投资风格是否长期稳定
- 是否存在为了短期排名偏离产品定位的迹象
- 是否发生了显著风格漂移

#### 为什么值得单列

因为 2026 年新落地的公募基金业绩比较基准指引，把“业绩比较基准”提高到了非常核心的位置：基准应体现产品投资目标和投资风格，帮助投资者判断预期风险收益特征，也要约束管理人的投资行为。

换句话说，基金评价如果不围绕 benchmark 做，很容易失焦。(csrc.gov.cn)

---

### Agent 6：费用 / 交易便利 / 持有期建议 agent

这个 agent 是本次需求中特别需要新增的。

#### 作用

输出以下内容：

- 费率结构评价
- 赎回成本
- 是否有最短持有期
- 交易便利性
- 流动性提示
- 持有期建议

#### 为什么重要

基金持有时长建议，不能只从收益波动出发，还要结合：

- 产品定位
- 策略周转率
- 赎回费安排
- 销售服务费安排
- 是否有锁定期 / 最短持有期
- 投资者风险承受能力

现行规则要求基金在招募说明书、基金合同等信息披露文件中载明申购、赎回价格计算方式和相关费率，并做好风险揭示；证监会 2026 年起施行的新《公开募集证券投资基金销售费用管理规定》还明确，对持有期限超过一年的基金份额（货币市场基金除外）不再收取销售服务费，并鼓励长期持有。(csrc.gov.cn)

#### 这个 agent 应该如何给出“建议持有时长”

建议不要输出一个拍脑袋的固定值，而是输出区间：

- 不建议持有：证据冲突或不适配
- 试持有观察 1–3 个月
- 建议持有 6–12 个月
- 建议持有 1 年以上
- 适合长期配置 3 年以上

并给出原因，例如：

- 产品定位偏长期成长
- 赎回费 / 销售服务费结构鼓励长期
- 风格漂移小
- 经理稳定
- 历史回撤可接受

---

### Agent 7：基金事件 / 变更 / 风险公告 agent

这是从原来的新闻 agent 演化出来的。

#### 作用

跟踪并解析以下事项：

- 基金经理变更
- 分红公告
- 暂停申购 / 赎回
- 清盘预警
- 基金合同修订
- 招募说明书更新
- 基准调整
- 侧袋机制
- 流动性风险事项
- 管理人重大合规事件

#### 为什么需要

基金场景里的“事件”比个股新闻更结构化，也更影响持有建议。基金规则明确强调高风险产品要加强风险揭示，流动性风险工具、侧袋机制等事项都可能实质影响投资者体验和持有策略。

---

### Agent 8：基金报告产出 agent

#### 作用

把前面 7 个基金分析 agent 的输出结果进行统一汇总、冲突消解、优先级排序和自然语言表达，生成最终的基金分析报告。

报告既要保留专业分析深度，又要让用户能快速读懂：

- 值不值得关注
- 适不适合自己
- 建议持有多久
- 主要风险在哪里

#### 为什么必须单列

因为前面 7 个 agent 更像“专业分析模块”，它们负责在各自维度上给出判断；但真实用户最终需要的不是 7 份分散结论，而是一份：

- 结构完整
- 表达统一
- 逻辑一致
- 可解释
- 可追踪来源

的最终报告。

如果不单列报告 agent，而让每个 agent 自己写报告，通常会出现：

- 风格不统一
- 证据重复
- 结论互相打架
- 难以维护版本
- 难以控制篇幅与模板

所以报告产出 agent 最好作为最终收口层。

#### 输入建议

报告 agent 不直接消费原始行情或原始文档，而是消费前 7 个 agent 产出的结构化中间结果，主要包括：

- 产品定位与基金基础属性
- 业绩与风险指标
- 组合结构与持仓穿透结果
- 基金经理与管理人分析
- 基准一致性与风格漂移判断
- 费用、流动性与持有期建议
- 基金事件、变更与风险公告
- 总分、分项分、适配性结论、风险 flags、置信度

#### 输出建议

```json
{
  "report_title": "XX基金分析报告",
  "investment_view": "可关注/谨慎关注/不建议配置",
  "total_score": 82,
  "holding_period_suggestion": {
    "label": "建议持有1年以上",
    "reasons": [
      "产品定位偏中长期配置",
      "风格稳定性较高",
      "费用结构对长期持有更友好"
    ]
  },
  "executive_summary": "...",
  "strengths": ["..."],
  "risks": ["..."],
  "sections": [
    {"title": "产品定位与策略", "content": "..."},
    {"title": "业绩与风险表现", "content": "..."},
    {"title": "组合与持仓分析", "content": "..."},
    {"title": "基金经理与管理人", "content": "..."},
    {"title": "费用与持有建议", "content": "..."},
    {"title": "近期重要事件", "content": "..."}
  ],
  "disclaimer": "..."
}
```

### Agent 9：基金打分 agent

#### 作用

基金打分 agent 基于前面 7 个分析 agent 的结构化输出，对基金进行统一综合评分，并生成适合前端展示的：

- 分数卡
- 分项评分
- 优劣势标签
- 建议持有期限
- 适配性结论

它主要服务于用户快速判断“这只基金大致怎么样、适不适合进一步关注”。

#### 为什么必须单列

因为基金报告和基金打分虽然依赖相同的分析基础，但目标不同：

- 报告输出强调完整解释和深度阅读
- 打分输出强调快速判断和前端直观展示

如果让报告 agent 兼做打分，会导致：

- 打分接口过重
- 前端卡片渲染字段不稳定
- 评分规则难以单独维护
- 后续难以支持批量评分、基金排序和推荐列表

因此，基金打分 agent 应作为与报告 agent 并列的独立收口层。

#### 输入建议

基金打分 agent 不直接消费原始公告、净值或合同，而是消费前面 7 个分析 agent 的标准化中间产物，包括：

- 产品定位与基金基本属性
- 历史业绩与风险特征
- 持仓结构与风格暴露
- 基金经理与管理人稳定性
- 基准一致性与风格漂移情况
- 费用、流动性与持有期约束
- 近期事件、变更与风险提示
- 用户画像与前端请求上下文

#### 输出建议

基金打分 agent 的输出应优先服务于前端评分卡、榜单排序、摘要页展示和进一步跳转报告页，因此建议输出：

- 综合分
- 分项得分
- 投资观点标签
- 建议持有期限
- 核心优势
- 主要风险
- 适合 / 不适合人群
- 简短评分解释
- 前端展示所需的结构化字段

## 四、基金分析报告需要包含的内容

建议分成“必选模块”和“可选模块”。

### A. 必选模块

这些是标准基金报告几乎都应该有的。

#### 1. 报告封面信息

包括：

- 基金名称
- 基金代码
- 份额类别
- 基金类型
- 报告日期
- 数据截止日期

#### 2. 核心结论摘要

这是最重要的一页 / 第一段。

建议回答 5 个问题：

- 这只基金整体怎么看？
- 总分多少？
- 适合什么样的投资者？
- 建议持有多久？
- 主要风险是什么？

示例：

> 该基金整体表现中上，产品定位清晰，历史风险收益特征相对稳定，适合能接受中高波动、以 1 年以上为持有周期的投资者。需要关注其行业集中度较高及近期风格回撤。

#### 3. 总分与分项评分

建议展示：

- 总分
- 分项分
- 产品定位分
- 业绩风险分
- 组合结构分
- 经理团队分
- 基准一致性分
- 费用与交易分
- 事件风险分

同时要有简要解释：

- 哪些项拉高了分数
- 哪些项拖累了分数

#### 4. 建议持有时长

这部分必须单独成章，不要只在摘要里带一句。

建议包含：

- 建议持有区间
- 最短建议持有天数
- 更优持有窗口
- 给出建议的理由
- 不适合的持有方式

示例结构：

```text
建议持有时长：1年以上
不建议用途：短期博弈、资金高频进出
主要原因：
1. 产品定位偏长期权益配置
2. 历史净值波动较大，短期体验不稳定
3. 风格稳定性尚可，长期持有更容易体现策略效果
4. 费用结构更适合中长期持有
```

#### 5. 产品定位与策略分析

这一部分主要承接 `product_doc_agent`。

建议写清楚：

- 这是什么类型的基金
- 投资目标是什么
- 主要投资什么资产
- 风险收益特征如何
- 业绩比较基准是什么
- 产品定位是否清晰

这部分是整个报告的“产品定义层”。

#### 6. 历史业绩与风险表现

这一部分承接 `perf_risk_agent`。

建议包含：

- 历史收益表现
- 波动与最大回撤
- 风险调整后表现
- 相对基准表现
- 不同市场阶段表现
- 稳定性如何

这部分不要只列数字，要给结论，例如：

- “收益不错但回撤偏大”
- “中等收益但控制回撤能力较强”
- “超额收益存在但不稳定”

#### 7. 组合结构与持仓穿透分析

这一部分承接 `holdings_lookthrough_agent`。

建议包含：

- 资产配置结构
- 行业集中度
- 持仓集中度
- 风格暴露
- 前十大持仓质量
- 是否存在风格漂移
- 是否与产品定位匹配

如果以后要做很强的专业报告，这部分会是核心竞争力。

#### 8. 基金经理 / 团队 / 管理人分析

这一部分承接 `manager_team_agent`。

建议包含：

- 基金经理任职时间
- 管理风格是否稳定
- 历史同类产品表现
- 团队是否稳定
- 管理人平台能力

需要输出的不是履历罗列，而是判断：

- 值不值得因为“人”而加分
- 是否需要因为经理变更而谨慎

#### 9. 基准一致性与风格稳定性

这一部分承接 `benchmark_style_agent`。

建议包含：

- 是否符合其业绩比较基准定位
- 是否存在长期偏离
- 风格是否稳定
- 是否出现风格漂移
- 当前表现是“策略生效”还是“偏离定位换来的短期结果”

这一部分决定报告的专业感。

#### 10. 费用、流动性与持有体验

这一部分承接 `fee_liquidity_holding_period_agent`。

建议包含：

- 申购费、赎回费、销售服务费
- 是否有最短持有期
- 流动性限制
- 交易便利性
- 成本是否偏高
- 对持有期建议的影响

这部分是连接“分析”和“投资体验”的桥梁。

#### 11. 近期重要事件与风险提示

这一部分承接 `fund_event_agent`。

建议包含：

- 基金经理变更
- 分红
- 暂停申赎
- 合同修订
- 基准调整
- 风险公告
- 其他重大事项

这部分要强调“近期变化是否改变结论”。

#### 12. 适合人群与不适合人群

这是非常实用的一部分。

建议输出：

- 适合什么风险承受能力的投资者
- 适合什么持有周期
- 适合什么投资目的
- 不适合什么人群

例如：

- 适合：能接受净值波动、以 1 年以上配置为目的的投资者
- 不适合：追求短期套利、无法承受阶段性回撤的投资者

#### 13. 免责声明 / 方法说明

建议固定保留。

包括：

- 本报告基于历史数据与公开资料
- 不构成未来收益承诺
- 不构成个性化投资建议
- 部分持仓分析可能存在披露滞后
- 评分结果受数据完整性和时效性影响

这部分非常重要，工程上最好模板化。

---

### B. 可选模块

适合进阶版报告。

#### 1. 同类基金对比

- 同类平均水平
- 同类排名
- 同类中优势与短板

#### 2. 情景分析

- 若市场偏成长 / 偏价值 / 震荡 / 下行时基金可能表现如何

#### 3. 用户适配分析

- 如果有用户画像，可单独写“为什么适合你 / 不适合你”

#### 4. 组合配置建议

不是推荐买多少，而是说明适合作为：

- 核心配置
- 卫星配置
- 防守配置

## 八、基金打分 agent 的评分维度

建议仍然沿用前面 7 个 agent 的维度来打总分，这样口径最稳定。

### 1. 产品定位与策略清晰度

来源：`product_doc_agent`

看什么：

- 基金类型是否清晰
- 投资目标是否明确
- 投资策略是否稳定
- 产品定义是否自洽
- 份额类别和运作安排是否明确

这个维度解决的问题是：

- 这只基金到底“是什么”
- 是否名副其实

### 2. 业绩与风险表现

来源：`perf_risk_agent`

看什么：

- 历史收益表现
- 波动率
- 最大回撤
- 风险调整后收益
- 与基准相比的超额表现
- 稳定性和连续性

这个维度解决的问题是：

- 这只基金过去“做得怎么样”，但不把过去直接等于未来

### 3. 组合结构与持仓质量

来源：`holdings_lookthrough_agent`

看什么：

- 资产配置结构
- 行业集中度
- 重仓股集中度
- 风格暴露
- 与基金定位是否一致
- 是否存在异常偏离

这个维度解决的问题是：

- 这只基金实际上“买了什么”
- 组合质量如何

### 4. 基金经理与团队稳定性

来源：`manager_team_agent`

看什么：

- 基金经理任职时长
- 管理连续性
- 团队稳定性
- 平台能力
- 管理风格延续性

这个维度解决的问题是：

- 这只基金背后的人是否稳定、可信、可持续

### 5. 基准一致性与风格稳定性

来源：`benchmark_style_agent`

看什么：

- 是否符合业绩比较基准定位
- 是否长期偏离
- 风格是否漂移
- 偏离是否合理可解释

这个维度解决的问题是：

- 这只基金是否“说到做到”

### 6. 费用、流动性与持有体验

来源：`fee_liquidity_holding_period_agent`

看什么：

- 申购费、赎回费、销售服务费
- 是否有锁定期或持有约束
- 流动性是否便利
- 持有成本是否合理
- 是否适合长期拿、短期拿

这个维度解决的问题是：

- 这只基金“拿起来顺不顺手、值不值得拿”

### 7. 事件风险与近期变化

来源：`fund_event_agent`

看什么：

- 经理变更
- 分红
- 申赎暂停
- 合同修订
- 风险公告
- 近期重大事项是否影响原有判断

这个维度解决的问题是：

- 这只基金最近有没有发生会改变结论的事

## 九、打分权重设计

当前不区分短中长评分，因此权重建议走统一中性版。

### 推荐权重示例

```text
产品定位与策略清晰度          15%
业绩与风险表现                25%
组合结构与持仓质量            20%
基金经理与团队稳定性          15%
基准一致性与风格稳定性        10%
费用、流动性与持有体验        10%
事件风险与近期变化             5%
```

### 为什么这样分

因为：

- 基金评价里，业绩风险和组合质量通常权重最高
- 产品定位和经理团队是第二层关键因子
- 基准一致性决定可信度，但一般不需要比业绩更重
- 费用流动性重要，但通常不是决定基金质量的唯一核心
- 事件风险通常是修正项，因此权重较低，但遇到重大事项可以触发强修正

## 十、建议持有期限如何输出

这是本次需求中特别提到的重点。

### 需求前提

要求：

- 不区分短中长评分
- 但前端页面仍然要展示建议持有期限

这个设计是合理的。

### 建议持有期限的定位

建议持有期限不是一个独立评分维度，而是一个综合推导结果。

它应该来源于以下因素共同作用：

- 产品定位
- 波动与回撤
- 持仓风格稳定性
- 流动性与锁定期
- 费用结构
- 经理风格延续性
- 事件风险

### 推荐输出格式

#### 标签式

- 建议持有 6 个月以上
- 建议持有 1 年以上
- 建议持有 2 年以上
- 更适合长期配置

#### 数值式

- 最短建议持有：180 天
- 更优持有窗口：365~730 天

#### 场景式

- 不建议短期波段操作
- 不建议作为高流动性备用资金配置

### 持有期限的推导逻辑示例

#### 更偏短一些的基金

可能输出：

- 建议持有 6 个月以上

适用特征：

- 波动相对可控
- 流动性较高
- 风格稳定
- 无强锁定期
- 产品策略实现周期相对较短

#### 标准权益基金

可能输出：

- 建议持有 1 年以上

适用特征：

- 净值波动较明显
- 风格切换影响较大
- 组合价值释放需要时间

#### 高波动 / 高集中 / 风格鲜明产品

可能输出：

- 建议持有 2 年以上

适用特征：

- 波动和回撤明显
- 风格偏成长或偏主题
- 短期体验可能较差
- 长期持有更能反映策略效果

## 十三、共享 merge 节点设计

强烈建议增加一个共享的 merge 节点。

### 为什么必须有

因为现在第 8 个报告 agent 和第 9 个打分 agent 是并列的，如果没有 merge 节点，会出现：

- 两个 agent 都要重复做字段清洗
- 两个 agent 都要做冲突消解
- 两个 agent 都要做持有期限归纳
- 两个 agent 都要做优先级排序

这会导致系统维护成本大增。

### 推荐新增一个共享节点

节点名称：

- `fund_analysis_merge_node`

### 作用

把前 7 个 agent 的输出统一成一个标准的“基金分析包”，供 `report agent` 和 `scoring agent` 共用。

### merge 节点输出示例

```json
{
  "fund_profile": {},
  "normalized_subscores": {
    "product_positioning": 85,
    "performance_risk": 78,
    "portfolio_structure": 84,
    "manager_team": 80,
    "benchmark_style_consistency": 79,
    "fee_liquidity": 76,
    "event_risk": 83
  },
  "strengths_pool": [],
  "risks_pool": [],
  "conflicts": [],
  "holding_period_hints": [],
  "confidence_summary": 0.81,
  "missing_data_summary": [],
  "frontend_ready_tags": []
}
```

### 这样设计的结果

- `fund_report_agent` 用它生成长文本报告
- `fund_scoring_agent` 用它生成评分卡和摘要结果

## 十四、总结

本次“基金专区”能力建设，建议采用如下整体方案：

- 在现有智能投顾项目中新增与股票功能并列的基金功能模块
- 建立 **7 个基金分析 agent + 1 个报告 agent + 1 个打分 agent**
- 前 7 个 agent 并行运行
- 中间通过统一的 `fund_analysis_merge_node` 做归一化汇总
- 最终按前端请求进入：
  - `fund_report_agent`
  - `fund_scoring_agent`

这个方案的核心优势在于：

- 模块职责清晰
- 输出口径统一
- 可支持报告与打分双场景
- 便于后续扩展批量评分、排序、推荐列表和用户适配分析
- 对 Claude Code 这类工程实现型助手更友好，便于按模块逐步落地开发

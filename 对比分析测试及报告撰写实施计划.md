## 2. 阶段一：category 回填（任务一）

### 2.1 规模确认（dry-run）

```powershell
<venv-python> scripts/backfill_signal_pack_category.py
```

预期输出：`待回填文件: ~2,300+ 个，待归类信号: ~10,000+ 条`（2026-07-22 实测为 2,353 文件 / 10,639 信号，随新缓存产生会略增）。记录实际数字进报告附录。

### 2.2 小批量试跑 + 人工抽查

```powershell
<venv-python> scripts/backfill_signal_pack_category.py --apply --limit 20
```

- 检查写回的 20 个文件：每条 signal 应有合法 `category` 枚举值、`category_source: "v4flash_backfill"`、文件级 `_schema_version: 2`，其余字段（含 created_at）未被改动；
- 人工抽查其中 ~10 条信号的归类是否合理（对照枚举含义：fundamentals_growth/valuation/capital_flow/technical_trend/sentiment/catalyst_event 等）；
- 归类明显错误率 >20% → 停下来调整脚本 prompt（枚举释义细化）后再继续，不要全量跑。

### 2.3 全量回填

```powershell
<venv-python> scripts/backfill_signal_pack_category.py --apply --batch-size 10
```

- 预计 ~240 批，25~45 分钟，成本个位数元；
- 关注进度输出中的失败计数；失败文件保持原样（后续按 schema 版本 miss 自然重跑，属设计内兜底）。

### 2.4 覆盖率验证

```powershell
<venv-python> scripts/backfill_signal_pack_category.py   # 再次 dry-run
```

- 待回填文件应为 0 或仅剩失败文件；记录失败清单；
- 若失败文件占比 >5%，分析失败原因（LLM 输出格式/网络/超时），可对这些文件单独再跑一次 `--apply`（脚本幂等：已含 category 的 pack 不会重复处理）。

### 2.5 回填质量抽样验证（易遗漏点①，必做）

回填 category 是 LLM 判定的，需要验证其质量后再用于比对：

1. **原生 vs 回填对照**：07-22 之后新产生的缓存（如德明利 sz.001309，07-22）的 category 是 Agent 原生输出的（无 `category_source` 标记），是天然的"标准答案"。抽取 5~10 只同时具有原生 pack 和回填 pack 的股票，对比同类信号归类风格是否一致；
2. **抽样人工复核**：随机抽 20 个回填文件，人工核对归类准确率，目标 ≥85%；
3. 将验证结果写入报告"回填质量"一节。质量不达标 → 先修 prompt 重跑，再进比对。

## 3. 阶段二：离线比对（任务二 第一部分）

### 3.1 基线比对（默认策略 default5）

```powershell
<venv-python> scripts/offline_score_comparison.py --top-n 20
```

- 确认报告头部的 **category 覆盖率警告消失**（应接近 100%）；
- 产物：`data/analysis_reports/offline_comparison_<时间戳>.md/.json`。

### 3.2 多 strength 策略比对（易遗漏点②，4.9-11 决策依据，必做）

4.9-11 已实测：全部 14,437 条历史信号中 `direction≠0 且 strength=0` 占 **18.1%**（quality_risk 最差 35.7%、event 33.7%）。该字段是公式承重墙，需用三种策略各跑一次比对，为最终口径提供数据：

```powershell
$env:DETERMINISTIC_SCORER_STRENGTH_POLICY="raw";                <venv-python> scripts/offline_score_comparison.py --top-n 20 --out data/analysis_reports/offline_cmp_raw.md
$env:DETERMINISTIC_SCORER_STRENGTH_POLICY="default5";           <venv-python> scripts/offline_score_comparison.py --top-n 20 --out data/analysis_reports/offline_cmp_default5.md
$env:DETERMINISTIC_SCORER_STRENGTH_POLICY="confidence_scaled";  <venv-python> scripts/offline_score_comparison.py --top-n 20 --out data/analysis_reports/offline_cmp_confscaled.md
Remove-Item Env:DETERMINISTIC_SCORER_STRENGTH_POLICY
```

- 对比三组指标差异：哪个策略的 Spearman 更高、MAE 更低，就支持哪个口径；
- 报告中必须给出明确倾向性结论（这是 4.9-11 的最终输入）。

### 3.3 指标体系（脚本自动产出 + 需人工补充计算的部分）

脚本已产出（按期限 short/medium/long）：

- Spearman 排序相关、MAE、均值差、Top-20 重叠率；
- strength 分布表（按 Agent 分组）；
- 偏差最大的 20 条记录明细；
- 历史报告矛盾率审计（4.9-2）。

需人工（或临时脚本）补充的分析：

1. **评级迁移矩阵**：LLM 评级（强烈买入/买入/中性/减持/回避）× 确定性评级（同档映射，4.9-4 阈值表）的 5×5 混淆矩阵，验证评级阈值是否需要校准；
2. **维度级一致性**（易遗漏点③）：LLM scorer 缓存 JSON 内含 `sub_scores`（各维度分），确定性输出也有 `sub_scores`——按期限逐维度对比均值差与相关，定位"哪个维度分歧最大"（通常直接指向维度-category 映射或权重问题）；
3. **风险闸门一致率**：双方 risk_gate 触发（cap/abstain/flags）的重合率，目标 ≥95%；
4. **分层一致性**（易遗漏点④）：
   - 按 native-category 样本（07-22 后新缓存）vs backfill 样本分组——检验回填质量是否拖低一致性；
   - 按股票 vs ETF 分组——验证 ETF 期望缺席归一化（4.9-3）是否正确（ETF 不应被缺失惩罚压到 ≤40）；
   - 按 coverage=1.0 vs <1.0 分组——区分"证据完整时的公式分歧"与"证据缺失时的惩罚分歧"。

### 3.4 验收门槛（沿用定稿）

| 指标 | 门槛 |
|---|---|
| Spearman 排序相关 | ≥ 0.9 |
| Top-20 池重叠率 | ≥ 85% |
| 风险闸门触发一致率 | ≥ 95% |
| 均值差绝对值 | ≤ 3 分 |

达标 → 进入阶段五（报告）与阶段六（切换建议）；不达标 → 进入阶段四（校准迭代）。

## 4. 阶段三：差异归因深挖（任务二 第二部分：从日志探究差异）

对每期限取 |diff| 最大的 5~10 只股票（取自比对 JSON 明细），逐只做归因分析：

1. **委派 log-reviewer**（必须，全局政策）：提供股票代码、日期、双方分数，让其从 `logs/` 中还原该股票 LLM scorer 的输入（AnalysisPackage 摘要、行业识别、缺失声明）与推理要点（llm_interactions 留痕在 `logs/<批次>/llm_interactions/`）；
2. **主线对照确定性 scorer 的中间量**：该股票的维度分、信号归组、冲突折扣、缺失惩罚，定位差异落在哪个环节；
3. **归因分类**（报告差异章节按此框架组织）：
   - a. 维度-category 映射分歧（信号进了不同维度）；
   - b. strength 字段质量（0 值/量纲不一致导致的贡献丢失）；
   - c. 中性信号与文本信息差（LLM scorer 读全文，确定性只读 direction≠0 的结构化信号）；
   - d. 行业识别错误干扰（易遗漏点⑤：已证实 `identify_industry` 字典序首匹配 bug 会让"证券"1 票胜"半导体"10 票，德明利被误判为非银金融——历史 LLM 分数本身可能是在错误行业框架下打的，属于**基准数据污染**，比对时需单独标注受影响股票，不计入公式分歧）；
   - e. Agent 结构化产出兜底（如 value agent 0 信号 text fallback，证据缺失型分歧）;
   - f. 技术指标预计算失败（2026-06-03 至今的既有问题，短线 LLM 分数中 MACD/RSI 为估算值，属基准噪声）；
   - g. 冲突折扣规则差异（LLM 的自由裁量 vs 代码 source_level 规则）。
4. **日志分析注意事项**（易遗漏点⑥）：`logs/src.stock_pool.scoring_engine.log` 中 07-22 14:56/15:00 有两条"测试股份 short评分完成: 59.0 分"记录，那是部署后 e2e 测试的产物（scorer_type=deterministic），分析日志时忽略之；`测试股份/sh.999999` 不是真实股票。

## 5. 阶段四：校准迭代（条件分支：门槛不达标时执行）

按以下顺序逐项调整并重新比对，每轮记录改动与指标变化（权重版本依次 det_v2、det_v3…）：

1. **先查归因分布**：若差异主要来自 d/e/f（基准数据污染），则**不是公式问题**——在报告中如实说明，不应为迁就污染基准而改公式；
2. **调 DIMENSION_SCALE**（默认 2.0，env `DETERMINISTIC_SCORER_DIMENSION_SCALE`）：若确定性分数分布明显比 LLM 更集中/更分散，先调缩放使分布对齐；
3. **调维度-category 映射**：若归因集中在 a 类（参考阶段二 3.3-2 的维度级对比），修正 `TERM_DIMENSIONS` 的 categories（4 轮审查已修过一次长期维度 industry_policy/ownership 重叠与中期 capital_flow 缺口，可能还有残留）；
4. **最后才动权重表**：权重是业务语义，改动需在报告中逐条说明理由；
5. 每轮迭代上限建议 3 次，仍不达标则在报告中如实记录并给出"暂不切换"的建议。

## 8. 易遗漏点汇总（用户要求专项核对，全部已纳入上文对应步骤）

7. **07-14 `.env` 密钥清空事件**：开工前必须验证密钥；数据窗口有效性需在报告中说明——见 §1、§6-2；

## 11. 风险与注意事项

- 回填脚本写盘采用临时文件 + 原子替换，中断不产生半文件；回填只新增字段、不改 created_at/TTL，不延长缓存寿命；
- 回填失败的文件会按 schema 版本 miss 在生产中自然重跑（设计内兜底），报告中列出清单即可，无需强制补齐；
- 比对脚本只读缓存目录，不修改任何生产数据；
- 本计划全程不要求重启 app/worker；若阶段六需要开 flag，届时再单独安排重启与验证；
- 若执行中发现缓存被清理或密钥再次失效，立即停止并回报用户。

## 二、发现的问题（按严重度排序）

### P1-1 行业识别错误：德明利被误判为"非银金融"

- **现象**：德明利（半导体股）三个期限 scorer 的输入与输出均按"非银金融"行业框架打分，而 fundamental Agent 正确识别为半导体。
- **证据**：`logs/20260722_224259_54f23f2d/agents/short_term_scorer_execution.json`（detected_industry=非银金融）；`logs/src.agents.medium_term_scorer.log:68906`、`long_term_scorer.log:68106`。
- **根因**：[industry_knowledge.py](Finance/Financial-MCP-Agent/src/utils/industry_knowledge.py) 的 `identify_industry()` 采用"按 INDUSTRY_KEYWORDS 字典序首次命中即返回"。字典中"非银金融"（关键词含 证券/券商，第 606 行）排在"电子"（关键词含 半导体/芯片，第 608 行）之前；德明利 news 分析文本中"证券"出现 1 次、"半导体"出现 5 次、"芯片"出现 5 次，但首次匹配机制让 1 次"证券"胜出。
- **影响**：凡分析文本提及券商/证券/保险/信托/期货的股票都可能被误判，LLM scorer 会用金融股估值框架给非金融股打分，影响全部三个期限的行业化打分指引。
- **性质**：既有问题（非 4.1–4.4 改动引入）。
- **建议修复**：将 `identify_industry` 从"字典序首次匹配"改为"按关键词命中次数计分、取最高分行业"（德明利：电子 10 票 vs 非银金融 1 票 → 正确识别为电子）；或从非银金融关键词中移除过于通用的"证券/券商"并要求公司名命中。改动范围小、回归风险低。

### P1-2 技术指标预计算持续失败，MACD/RSI/均线退化为 LLM 估算

- **现象**：technical Agent 预计算阶段报错 `技术指标预计算失败: Expecting value: line 1 column 1 (char 0)`，导致 MACD/RSI/均线无精确数值，短线打分 data_reliability 被降为"中"。
- **证据**：`logs/src.agents.technical_agent.log:198128`（德明利，2026-07-22 22:43:02）。
- **根因**：`tushare_kline` 工具返回格式与 `_build_indicator_summary()` 的解析逻辑不匹配（JSON 解析失败）。
- **性质**：既有问题——日志显示该报错自 **2026-06-03** 起持续出现（`logs/src.agents.technical_agent.log:3859`），与 4.1 的 prefetch 钩子无关（已用历史日志交叉验证）。
- **影响**：短线打分的核心技术指标长期依赖 LLM 手工估算，精度和一致性受损；也是后续 4.6 确定性特征层的前置障碍。
- **建议修复**：核对 `tushare_kline` 实际返回结构，对齐 `_build_indicator_summary` 的解析（或在解析失败时明确降级声明而非静默 WARNING）。

### P2-1 value Agent 结构化产出失败（本次德明利个例）

- **现象**：value_analysis 的 signal_pack 落入文本兜底（`missing_data=['结构化产物缺失，已从文本推断']`），0 条信号、置信度 30%。
- **证据**：`data/intermediate_cache/value_analysis_signal_pack_sz_001309_2026-07-22.json`。
- **根因**：value Agent 的 LLM 本次未输出合法 SIGNAL_PACK JSON，三层提取（tag→regex→text fallback）落到最后一层。
- **性质**：既有 LLM 输出合规性问题（非 4.1–4.4 改动引入）；打分未中断（其余 6 个 Agent 提供了信号），但估值维度证据缺失。
- **建议**：随 4.3 离线比对观察该类 fallback 的发生频率；若频发，在模型输出边界加一次结构化修复重试。

### P2-2 DataGateway 预取命中率在异常窗口期偏低

- **现象**：德明利预取命中 19/32；当天密钥失效窗口期其他标的普遍 2/32，sz.002241 为 0/32，未见对未命中项的补偿取数日志。
- **证据**：`logs/src.data.data_gateway.log:2160-2181`。
- **性质**：按设计静默降级——预取是优化不是闸门，未命中项由各 Agent 直取补齐（德明利 7 个 Agent 均成功产出），无实际数据缺失；低命中主要与当时 `.env` 密钥失效/MCP 未就绪的环境窗口叠加有关。
- **建议**：无需立即处理；可在预取日志中补充未命中项清单，便于区分"工具不可用"与"取数失败"。

## 1. 系统定位与边界

### 1.1 一句话定义

评分智能体不是第二个荐股系统，而是现有A股/基金智能投顾系统**外面的一层评估控制塔**，用于：手动触发全链路检查、基于真实市场结果结算Loss、对各Agent做消融贡献归因、生成评测报告、对简单问题安全自动优化、积累长期记忆并展示趋势。

### 1.2 核心价值

1. 现有系统的推荐结果是否真实有效
2. 短线/中线/长线/基金等不同能力分别表现如何
3. 哪个分析Agent在帮忙，哪个在拖后腿
4. 迭代到底有没有改善（版本→Loss趋势）
5. 哪些问题可以自动修，哪些必须人工处理
6. 让迭代真正围绕"真实市场结果"进行

### 1.3 V1 明确不做

- 自动定时运行全量检查（必须人工触发或每日手动一键）
- 高频分钟级策略评测
- 复杂架构自动重构
- 新Agent自动设计与自动接入
- 复杂Prompt/架构/数据源问题的自动修改
- 替代现有主投顾系统的核心功能
- 不引入AkShare新依赖，不引入Wind/iFinD等昂贵数据库
- 不将自动优化结果直接合并到stable主分支

## 2. 整体架构

### 2.1 核心流程

```
┌─────────────────────────────────────────────────────────┐
│                    评分智能体完整架构                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ═══ 低频操作（数周/数月一次，用户手动触发）══════          │
│                                                         │
│  全A股 (~5000只)                                         │
│    ↓ 纯代码去ST/*ST/新股未满60天/日均成交额不足             │
│  ~4500只                                                 │
│    ↓ 本项目正式模型(M1/M3) 批量打分（短/中/长线各自独立）     │
│    ├── "强烈推荐" → 白名单 → 直接进入精筛池候选             │
│    ├── "买入/谨慎买入/观望" → 初筛池 → 快筛 → 达阈值进入     │
│    └── "卖出" → 黑名单（120天有效期）                      │
│    ↓                                                     │
│  白名单 + 初筛通过(1:1.2) → 正式模型精筛 → 去掉后20%        │
│    ↓                                                     │
│  三个精筛池各自确定，长时间不变                             │
│                                                         │
│  ═══ 高频操作（日常，用户手动一键触发）══════════            │
│                                                         │
│  模拟盘（14条线）:                                         │
│    用户点击"一键完整检查" → 自动检测上次结算日 → 追赶所有     │
│    错过的交易日（逐日：拉历史数据→评分调仓→收盘结算）         │
│    短线10条线：每日调仓 | 中线2条：每周五 | 长线2条：每月末   │
│                                                         │
│  历史回测（23条线，用户手动触发）:                           │
│    短线7条(6消融+1长持对照) | 中线8条 | 长线8条              │
│                                                         │
│  ═══ 分析与优化 ═══════════════════════════              │
│                                                         │
│  Loss引擎 → 归因引擎 → LLM诊断 → 优化路由 → 自动/人工      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 两大核心闭环

**快反馈（回测）**：历史数据重放，几十分钟内得到Agent贡献结论，用于快速迭代方向指引。

**真反馈（实盘）**：每天真金白银模拟，等待真实市场验证，用于最终确认。

快反馈决定"往哪个方向改"，真反馈决定"改对了没有"。

## 3. 线路结构总表

### 3.1 总计：14条实盘线 + 23条回测线 = 37条（回测中2条为退化线，1条为长持对照线，实际有效34条消融线）

### 3.2 短线（10条实盘 + 7条回测：6消融+1长持对照）

| 线路 | 类型 | Agent配置 | 策略 | 调仓频率 | 说明 |
|------|------|-----------|------|---------|------|
| S-L0 | 实盘消融基准 | 全部7 agent | 极简日清策略 | 每天 | 交叉截面锚点，每天开盘前同步所有消融线持仓 |
| S-L1 | 实盘消融 | 去掉fundamental | 极简日清策略 | 每天 | 从S-L0持仓同步出发 |
| S-L2 | 实盘消融 | 去掉technical | 极简日清策略 | 每天 | 同上 |
| S-L3 | 实盘消融 | 去掉value | 极简日清策略 | 每天 | 同上 |
| S-L4 | 实盘消融 | 去掉news | 极简日清策略 | 每天 | 同上 |
| S-L5 | 实盘消融 | 去掉event | 极简日清策略 | 每天 | 同上 |
| S-L6 | 实盘消融 | 去掉quality_risk | 极简日清策略 | 每天 | 同上 |
| S-L7 | 实盘消融 | 去掉moneyflow | 极简日清策略 | 每天 | 同上 |
| S-L8 | 实盘长持 | 全部7 agent | 成熟短线策略 | 每天 | 连续持仓运作，不复位 |
| S-L9 | 实盘LLM自由 | 无agent | LLM自主决策 | 每天 | DeepSeek V4，直接调用MCP，不用任何agent中间产物 |

**短线消融线交叉截面机制**：
- 每天开盘前，S-L0至S-L7全部同步为S-L0前一日的收盘持仓
- 然后各线独立评分、独立买卖
- 当日收益差异直接归因到Agent配置差异，消除历史持仓路径依赖
- 消融线采用极简策略（日清式，每天从同一起点出发只取Top10），这是**实验设计上的极端情况**，不代表真实交易方式
- S-L8（长持线）才是正常短线交易模式，作为消融实验的"现实参照系"

**短线回测消融（6条 + 1条长持对照线，用户手动触发）**：

| 线路 | Agent配置 | 策略 | 说明 |
|------|-----------|------|------|
| SB-L0 | 全部可回溯agent（不含news/event） | 极简日清 | 回测基准 |
| SB-L1 | 去掉fundamental | 极简日清 | |
| SB-L2 | 去掉technical | 极简日清 | |
| SB-L3 | 去掉value | 极简日清 | |
| SB-L4 | 去掉quality_risk | 极简日清 | |
| SB-L5 | 去掉moneyflow | 极简日清 | |
| SB-L6 | 全部可回溯agent（不含news/event） | 成熟短线策略（连续持仓，不每天清仓） | **长持对照线，不参与消融ΔLoss计算** |

> news_agent和event_agent在回测中禁用（历史数据不可回溯），因此短线回测只做5个Agent的消融（+1基准=6条消融线）。
>
> **SB-L6 长持对照线**：使用与模拟盘 S-L8 相同的成熟短线策略（连续持仓、多信号保护），在回测的相同 anchor dates 上运行。目的：作为消融实验的"现实性校验"——消融实验说"去掉 technical agent 导致 Loss 增加 0.04"，这个结论在连续持仓的真实场景下是否依然成立？SB-L6 提供交叉验证。SB-L6 的收益序列不与消融线直接做 ΔLoss 计算，而是作为独立参照系展示在回测报告中。

### 3.3 中线（2条实盘 + 8条回测消融）

**中线实盘不做消融实验**（等待周期太长，依赖回测）。

**中线回测消融（8条，用户手动触发）**：

| 线路 | Agent配置 | 说明 |
|------|-----------|------|
| MB-L0 | 全部可回溯agent（不含news/event） | 回测基准 |
| MB-L1 | 去掉fundamental | |
| MB-L2 | 去掉technical | |
| MB-L3 | 去掉value | |
| MB-L4 | 去掉quality_risk | |
| MB-L5 | 去掉moneyflow | |

### 3.4 长线（2条实盘 + 8条回测消融）

| 线路 | 类型 | Agent配置 | 策略 | 调仓频率 |
|------|------|-----------|------|---------|
| L-L0 | 实盘全agent | 全部7 agent | 长线深度价值策略 | 每月末 |
| L-L1 | 实盘LLM自由 | 无agent | LLM自主决策 | 每月末 |

**长线回测消融（8条，用户手动触发）**：中线回测的8条线配置，唯一区别是持有期和策略参数。

### 3.5 LLM自由线约束

三条LLM自由线（S-L9、M-L1、L-L1）的统一约束：
1. **可用**：本项目MCP工具（Tushare全部接口），直接访问原始市场数据
2. **不可用**：任何分析Agent的中间产物（`{agent}_analysis`、`{agent}_signal_pack`、`analysis_package`）
3. **不可用**：任何scorer的输出（含评分、排名、推荐动作）
4. **不可用**：任何Agent产出的评分（`score`字段），无论是否标注"仅供参考"——LLM自由线prompt中不得出现Agent评分，LLM必须仅基于原始市场数据自主判断
5. LLM自由线的目的是**对照**：一个有能力的基模，不经过Agent体系，能否比Agent系统做得更好？
6. LLM自由线的交易同样经过市场仿真层（佣金/滑点/成交量/停牌/涨跌停），与其他线公平对比
7. **模型**：DeepSeek V4或DeepSeek V4.1

## 4. 精筛股票池筛选流程

### 4.1 筛选总流程

```
全A股 (~5000只)
  ↓
[第0层：纯代码硬筛 — 无LLM调用]
  ├── 剔除 ST 和 *ST
  ├── 剔除 上市不满60天
  └── 剔除 近20日日均成交额 < threshold（可配置，默认2000万）
  → 剩余 ~4500只
  ↓
[第1层：正式模型批量打分 — 使用本项目生产模型M1/M3]
  对短/中/长线各自独立打分（需分别跑3次批量打分）
  打分结果分为4档：
  ├── "强烈推荐" → 白名单 → 直接进入精筛池候选（跳过初筛）
  ├── "买入"     → 进入初筛池
  ├── "谨慎买入" → 进入初筛池
  ├── "观望"     → 进入初筛池
  └── "卖出"     → 黑名单（120天有效，过期重新评估）
  ↓
[第2层：初筛池快筛 — 使用快筛模型]
  对初筛池股票跑快筛打分（便宜模型，更快但精度略低）
  快筛分数 < 阈值 → 淘汰
  快筛的目的是"去掉明显不行的"，不是"选出最好的"，阈值不宜过高
  ↓
[第3层：精筛池定稿]
  白名单股票 + 初筛通过股票 混合
  按 白名单:初筛通过 ≈ 1:1.2 的比例组建精筛池候选
  （若白名单实际数量不足，差额由初筛通过中排名靠前的补上）
  （若白名单实际数量超出，按正式评分取前N名）
  ↓
  精筛池候选用正式模型重新打一次完整分
  按绝对阈值（非固定比例）淘汰低分股票：
    - 阈值由内置LLM根据得分分布动态设定
    - 波动范围：使筛选后股票数在目标池大小的90%~110%之间
    - 最终各期限精筛池大小：
      · 短线精筛池：目标100只
      · 中线精筛池：目标80只
      · 长线精筛池：目标60只
```

### 4.2 精筛池更新触发条件

```
用户手动触发（前端按钮）+ 内置自动检测 + 前端红点提示

用户每次打开项目时，系统自动检查以下条件，无需等待用户主动查看：
  满足以下任一条件时，前端精筛池区域显示红点（严重）或黄点（提醒）：

  🔴 红点（建议尽快更新）：
  1. 池内超过30%的股票连续5个交易日评分 < 50
  2. 某条LLM自由投资线持续跑赢所有Agent线超过配置幅度
  3. 黑名单股票出现重大基本面变化

  ⚠ 黄点（临近阈值，建议关注）：
  4. 池内超过20%的股票连续10个交易日未被任何线持仓
  5. 距离上次全量更新超过配置天数：
     短线：30天 | 中线：60天 | 长线：90天

  正常状态显示绿点，表示精筛池健康

检测逻辑位于 pool_manager.py 的 needs_update() 方法中（纯代码，不调LLM），
前端渲染时调用并展示视觉标记。用户点击红点/黄点可查看详情和建议操作。

更新类型：
  - 部分更新：只替换评分持续垫底的股票（后20%）
  - 全量更新：重新跑完整筛选流程
```

### 4.3 精筛池筛选的模型使用

> **重要区分**：精筛池筛选使用本项目**生产模型**（M1/M3），保证池子质量。
> 日常模拟盘中的分析Agent统一使用**MiMo-V2.5**，兼顾成本。
> 这是"好模型选池，便宜模型跑分"的分层策略。

## 5. 交易策略规范

### 5.1 策略总览

| 策略名 | 适用线路 | 调仓频率 | 持仓数 | 单只上限 | 最低现金 | 核心哲学 |
|--------|---------|---------|--------|---------|---------|---------|
| 极简日清 | S-L0~S-L7 | 每天 | 10 | 10% | 0% | 评分驱动，日清式，为消融实验设计 |
| 成熟短线 | S-L8 | 每天 | 12 | 12% | 10% | 连续持仓+多信号保护 |
| 中线价值+趋势 | M-L0 | 每周五 | 8 | 18% | 15% | 基本面+估值+趋势确认 |
| 长线深度价值 | L-L0 | 每月末 | 5 | 25% | 20% | 集中+深度+耐心+越跌越买 |
| LLM自主 | S-L9/M-L1/L-L1 | 对应频率 | LLM自定 | 无硬约束 | 无硬约束 | 基模自主决策 |

> 所有策略均为纯代码实现，不调用LLM做买卖决策。LLM自由线除外。

### 5.2 极简日清策略（短线消融线专用）

**目的**：将策略差异降到最低，让评分差异成为唯一的收益驱动因素。

```
每天开盘前（使用前一日收盘价模拟）：
  1. 从短线精筛池100只中，取评分最高的10只（且score ≥ 60）
  2. 等权买入，每只10%仓位
  3. 当日开盘价成交，收盘价计算收益
  4. 次日全部同步为S-L0持仓后，重新评分，重新选前10只
```

### 5.3 成熟短线策略（S-L8 短线长持线）

**目的**：模拟一个成熟的短线交易者，评分是核心但非唯一要素。

```
【选股层】
  买入条件（全部满足）：
    1. score ≥ 60
    2. 近5日日均换手率 ≥ 0.5%（流动性过滤）
    3. 非当日涨幅 > 8%的股票（不追高）
    4. 非连续2日跌停后首日（不接飞刀）
  
  买入排序：按 (score × 0.6 + 近5日动量 × 0.3 + 换手率分位 × 0.1) 降序

【持仓层】
  最大持仓：12只 | 单只上限：12% | 最低现金：10%

  买入规则：
    - 新开仓：分2天建仓（T日50%，T+1日50%），减少择时风险
    - 加仓：score > 75且近3日涨幅 < 8% → 额外加2%
    - 补仓：score > 55且近3日跌幅 > 5% → 补1%

  卖出规则：
    - 硬止损：score < 35 → 全部卖出
    - 软止损：score 在 35-45 → 卖出60%，留40%观察
    - 利润保护：从最高点回撤 > 8% → 卖出50%
    - 时间止损：连续5天score低于买入时score → 全部卖出
    - 换手率异常：单日换手率 > 25% → 卖出70%

【仓位权重】
  基础仓位 = 1/12 ≈ 8.3%
  调整因子 = (score - 50) / 50（score=75 → 1.5x，score=45 → 0.9x）
  最终仓位 = 基础仓位 × min(调整因子, 2.0)

【风控层】
  - 同行业不超过总资金35%
  - 收盘前30分钟异常波动(>3%)触发标记
  - 短线score > 70但中线score < 40 → 仓位减半（短中矛盾）
```

### 5.4 中线价值+趋势策略（M-L0）

```
【选股层】
  买入条件（全部满足）：
    1. score ≥ 65
    2. PE(TTM) > 0 且 PE < 行业75分位
    3. ROE > 5% 或 营收增速 > 10%
    4. 股价 > MA60
    5. 无 delist_risk / regulatory_risk
  
  买入排序：按 (score × 0.5 + ROE分位 × 0.2 + PE倒数分位 × 0.2 + 近20日动量 × 0.1)

【持仓层】
  最大持仓：8只 | 单只上限：18% | 最低现金：15%

  买入规则：
    - 新开仓：分4天建仓（每周调仓日建25%）
    - 加仓：score上升 ≥ 5分且PE分位下降 ≥ 10% → 加仓到上限
    - 越跌越买（有限）：score > 70且股价较建仓价跌 > 10% → 加仓30%
      （前提：基本面没有恶化，即fundamental_agent未检出earnings_quality_concern）

  卖出规则：
    - 核心卖出：score < 45
    - 估值卖出：PE超过行业90分位且score < 55
    - 趋势卖出：股价跌破MA60且3日未收回
    - 最大回撤：从最高点回撤 > 15% → 卖出50%

【跨期限信号协同】
  - 中线score > 70且长线score > 70 → "双重确认"，仓位上限提到22%
  - 中线score > 60但长线score < 35 → "短中期看好但长期堪忧"，仓位上限降到10%
```

### 5.5 长线深度价值策略（L-L0）

```
【选股层】
  买入条件（全部满足）：
    1. score ≥ 70（高确定性要求）
    2. ROE > 8% 且 连续3年净利润增长（或至少2年正+1年平）
    3. PE < 行业中位数 或 PB < 行业中位数
    4. 无 delist_risk / audit_risk / regulatory_risk
    5. 股息率 > 0（至少分过红）
  
  买入排序：按 (score × 0.4 + ROE × 0.25 + 1/PE × 0.2 + 股息率 × 0.1 + 近60日反转 × 0.05)

【持仓层】
  最大持仓：5只 | 单只上限：25% | 最低现金：20%

  买入规则：
    - 一次性建仓50%，分2个月建完
    - 加仓：score上升 ≥ 8分 → 加仓到上限
    - 核心规则（越跌越买）：score > 75且股价跌 > 15% → 加仓50%

  卖出规则（非常克制，需要强证据）：
    - 基本面卖出：score < 35 持续2个月以上 → 分批卖出
    - 质量恶化：ROE连续2季下降 > 30% → 全部卖出
    - 估值极端：PE突破历史95分位 → 卖出30%
    - 黑天鹅：delist_risk / audit_risk → 立即清仓
    - 除此之外，不设止损，不因价格波动卖出
```

### 5.6 策略参数配置化

所有策略中的数值参数（阈值、权重、持仓数、比例等）均通过配置文件管理，不硬编码在策略类中。默认配置文件路径：`config/eval/strategy_defaults.json`。

## 6. 市场仿真层

### 6.1 定位

市场仿真层（`src/eval/market_simulator.py`）是模拟盘和回测**共用的基础设施**，统一处理所有现实交易约束。所有线路（包括LLM自由线）都经过同一仿真层，保证对比公平。

### 6.2 交易成本

```
A股实际交易成本：

  买入：
    佣金：成交金额 × 0.025%（万2.5），最低5元/笔
    过户费：成交金额 × 0.002%（万0.2）
    合计：约0.027%

  卖出：
    佣金：成交金额 × 0.025%（万2.5），最低5元/笔
    过户费：成交金额 × 0.002%（万0.2）
    印花税：成交金额 × 0.1%（千1，仅卖出）
    合计：约0.127%

  一次完整买卖总成本：约0.154%

滑点模拟：
  买入价上浮0.05%（模拟市场冲击）
  卖出价下浮0.05%
  大盘蓝筹股（沪深300成分股）滑点降为0.02%
  小微盘股滑点提升为0.1%
```

### 6.3 成交量约束

```
单笔买单金额 ≤ 该股当日成交额的5%
单日总买入金额 ≤ 该股当日成交额的10%

若因成交量约束无法完全成交：
  1. 记录为"流动性受限"
  2. 按最大可成交量执行
  3. 剩余资金按评分顺序投入下一候选
```

### 6.4 涨跌停约束

```
涨停（买入端）：
  收盘价 == 涨停价 → 当日无法买入，挂单失败
  盘中触及涨停但未封板 → 50%概率成交

跌停（卖出端）：
  收盘价 == 跌停价 → 当日无法卖出，持仓冻结
  持续跌停 → 每日尝试卖出，直到开板

T+1约束：
  当日买入的股票，次一交易日才能卖出
  （自然满足于收盘价模拟，买入次日才可卖）
```

### 6.5 停牌处理

```
停牌首日：标记该持仓为"冻结"，不计入当日组合收益
冻结期：不计算价格变动
冻结超限：冻结超过20个交易日 → 按停牌前价格 × 0.95强制估值
复牌首日：恢复交易，按复牌当日真实价格结算

回测中：通过Tushare suspend_d接口获取历史停牌信息
```

### 6.6 数据结构

```python
class Order:
    stock_code: str
    direction: str          # "buy" | "sell"
    quantity: int           # 股数（100的整数倍）
    target_value: float     # 目标成交金额
    order_type: str         # "market" | "limit"

class ExecutedOrder(Order):
    actual_price: float
    actual_value: float
    commission: float
    transfer_fee: float
    stamp_tax: float        # 仅卖出
    net_cost: float
    status: str             # "filled" | "partial" | "rejected"
    reject_reason: str      # "停牌" | "涨停" | "跌停" | "流动性不足" | ""
```

## 7. 模拟盘日常运行

> **设计原则**：模拟盘不追求实时性，追求"每个交易日都有完整的调仓+结算记录"。由于本项目为本地部署，用户不定期启动，系统采用"启动时手动一键追赶"模式。所有交易均使用收盘价模拟，数据完全可以从 Tushare 事后拉取。

### 7.1 追赶式运行模式

```
用户随时启动项目 → 点击"一键完整检查" → 系统自动执行：

  1. 检测上次结算日期（last_settled_date）
  2. 若存在错过的交易日：
     a) 计算缺失交易日数量
     b) 若缺失超过7个交易日：放弃追赶，从最新交易日重新开始
        （last_settled_date 重置为最新交易日，清空所有线持仓状态，
         前端明确提示"缺失超过7天，已重置为最新交易日"）
     c) 若缺失≤7个交易日：按时间顺序逐日追赶
        · 每日流程：拉取Tushare历史数据 → 运行分析Agent → 评分调仓 → 收盘结算
        · news/event agent 在追赶时标记 pit_mode="partial"（历史新闻不可回溯，
          与回测中禁用逻辑一致，scorer自适应降权）
  3. 追赶至最新交易日 → 保存所有快照 → 生成评测报告
  4. 用户可立即进行其他操作（股票查询/基金分析等）

非交易日：自动跳过，不执行任何操作

用户不点击"一键完整检查"：模拟盘不做任何事情，不消耗Token

最大追赶天数：7个交易日（可配置）— 超过此上限则重置，防止追赶耗时过长
  且历史数据Token消耗不可控。
```

### 7.2 调仓节奏

```
所有线的调仓频率在追赶模式下保持不变——系统根据历史日期判断是否为调仓日：

  短线全部10条线（S-L0~S-L9）：每个交易日调仓
    - S-L0~S-L7 消融线：每日交叉截面同步 → 独立评分 → 极简日清策略调仓
    - S-L8 长持线：每日评分 → 成熟短线策略调仓（连续持仓，不清仓）
    - S-L9 LLM自由线：每日评分 → LLM自主决策

  中线2条线（M-L0, M-L1）：
    系统检测追赶日是否为周五 → 是则执行中线策略调仓
    若追赶跨越多个周五 → 逐周五执行

  长线2条线（L-L0, L-L1）：
    系统检测追赶日是否为月末最后一个交易日 → 是则执行长线策略调仓
    若追赶跨越多个月末 → 逐月末执行

周一全量跑（短线精筛池）：
  追赶时若经过周一 → 触发短线精筛池全量打分
  若追赶跨度大导致成本过高 → 只对最后一个周一执行全量跑（通过配置控制）
```

### 7.3 单次追赶流程（每日）

```
对每个需要追赶的交易日：
  1. 通过Tushare获取 as_of_date 的历史收盘数据（daily/daily_basic/moneyflow等）
  2. 各线运行分析Agent（分级频率+缓存共享）
     - 可回溯agent（fund/tech/value/quality/money）：正常从历史数据运行
     - 不可回溯agent（news/event）：标记 pit_mode="partial"，使用空数据占位
  3. 各线运行Scorer（独立调用，不合并）
  4. 各线策略引擎根据评分生成买卖订单
  5. 市场仿真层执行订单（使用历史收盘价+滑点模拟）
  6. 保存调仓快照（prediction_snapshot）
  7. 使用当日收盘价结算持仓市值和收益
  8. 更新Loss追踪

每次追赶完成后：
  - 更新 last_settled_date 为最新已结算交易日
  - 前端展示追赶摘要（追赶天数/跳过的非交易日/各线最新收益）
```

### 7.4 初始状态

- 每条线初始资金：100万元（可配置）
- 起始状态：全部空仓
- 首次运行：从精筛池中按各自策略选股建仓
- 若首次运行日非交易日：使用最近一个交易日的数据建仓

## 8. 回测引擎设计

### 8.1 回测定位

回测的核心目的是**快速测量Agent贡献**，而非精确计算策略历史收益。回测结论用于指导优化方向，最终效果由实盘验证。

### 8.2 回测参数

| 参数 | 中线回测 | 长线回测 | 长线加速版 |
|------|---------|---------|-----------|
| 区间 | 2022-01 ~ 2026-05 | 2020-01 ~ 2025-06 | 2020-01 ~ 2025-06 |
| Anchor间隔 | 每周五 | 每月末 | 每周一 |
| Anchor数量 | ~230个 | ~66个 | ~280个 |
| 持有期 | 20个交易日 | 240个交易日 | 60个交易日 |
| 消融线数 | 8条 | 8条 | 8条 |
| 有效消融维度 | 5个（fund/tech/value/quality/money） | 5个 | 5个 |
| 退化线 | news消融/event消融（=基准） | 同 | 同 |

### 8.3 回测执行流程

```
对每个anchor date:
  1. 通过Tushare获取 as_of_date 及之前的所有历史数据
  2. 动态重建该日的可投资股票池：
     - 从daily表获取该日有成交量的所有股票
     - 排除 list_date > anchor_date 的股票（当时未上市）
     - 排除当日停牌股票（suspend_d）
     - 标记ST股票
  3. 对该池子跑分析Agent（仅可回溯的5个：fund/tech/value/quality/money）
     禁用news和event（历史数据不可用）
  4. 跑8条消融线scorer
  5. 每条线按对应期限策略调仓
  6. 持有对应天数
  7. 计算每条线的持有期收益

回测使用方案A（固定精筛池）：
  - 使用当前精筛池中的股票做回测
  - 优点：成本可控（不需要每个anchor date重新筛选）
  - 缺点：存在幸存者偏差（排除了回测期间退市/暴雷的股票）
  - 缓解：agent贡献排序受偏差影响小（所有消融线面对相同偏差）
  - 量化：方案A结果中自动计算并展示当前精筛池成分股在回测区间内的实际退市/暴雷数量，让用户对偏差大小有量化感知（而非仅笼统声明）
  - 声明：回测结果界面必须明确标注此偏差
  
季度校准（方案B，每季度跑一次）：
  - 在每个anchor date动态筛选当时的精筛池
  - 无幸存者偏差但成本高
  - 用于校准方案A的结论
```

### 8.4 信息封闭性

```
数据层面：
  - 所有Tushare查询指定 trade_date / end_date ≤ as_of_date
  - 财务数据按公告可见日对齐（不按报告期直接穿越）
  - as_of_date进入缓存key，防止未来数据污染历史缓存

Agent层面：
  - news_agent和event_agent在回测中完全禁用
  - 回测消融只测5个可回溯agent（fund/tech/value/quality/money）

LLM先验知识：
  - 评估模型MiMo-V2.5训练数据截止约2024年中期
  - 回测区间在截止日期之前的部分信息封闭性较好
  - 回测区间在截止日期之后的部分LLM可能有先验信息
  - 缓解：先验知识对所有消融线对称，ΔLoss受污染程度低
  - 前端标注：显示模型训练数据截止日期，供用户选择回测时间参考
  - 回测system prompt增加约束："你只基于提供的数据打分，不要使用训练数据中的任何外部知识"
```

### 8.5 样本外验证（防过拟合）

```
回测区间三段划分：
  训练期：2020-01 ~ 2023-12（4年）  → 消融实验与优化
  验证期：2024-01 ~ 2025-06（1.5年）→ 验证优化方向
  测试期：2025-07 ~ 2026-06（1年）  → 最终确认，不可再基于此调优

优化纪律：
  1. 只在训练期上做消融实验和优化搜索
  2. 在验证期上确认Loss是否改善
  3. 训练期+验证期都改善 → 置信度提高，可采纳
  4. 测试期只跑一次确认，绝不基于测试期结果继续优化
  5. 一旦基于测试期优化，测试期就变成了训练期，需要新的测试期

保守优化原则：
  - 改善幅度必须 > 1个标准差（不是噪声）
  - 改动必须有逻辑上合理的解释（不只是数据挖掘）
  - 不做细粒度参数寻优（如"阈值从60调到58"），只做粗粒度方向性调整
  - 禁止基于回测的数十次微调直到验证期Loss变好

市场环境切片验证：
  对同一个优化结论，在以下子区间上分别验证：
    牛市期（如2024.9-2025.6）
    熊市期（如2022.4-2022.10）
    震荡期（如2023.1-2023.6）
  如果只在某些市场环境下有效，系统记录此边界
```

### 8.6 回测前端界面

用户可自主设定回测参数：
- 回测期限（短线/中线/长线）
- 回测起止日期
- 消融实验的Agent选择
- 策略参数覆盖（可选）

## 9. Loss函数体系

### 9.1 总体框架

每个功能模块有独立Loss。Loss分为三层：

```
L_total = w_effect × L_effect + w_stability × L_stability + w_efficiency × L_efficiency

默认权重（可配置）：
  w_effect = 0.75
  w_stability = 0.15
  w_efficiency = 0.10
```

每个模块总分映射：
```
Score_m = 100 × (1 - L_total_m)
```

### 9.2 效果Loss（L_effect）— 核心

```
L_effect = L_return（收益端）+ L_risk（风险端）+ L_structure（结构端）

各期限权重（可配置）：

  短线: w_return=0.45, w_risk=0.40, w_structure=0.15
  中线: w_return=0.50, w_risk=0.35, w_structure=0.15
  长线: w_return=0.55, w_risk=0.30, w_structure=0.15
```

#### 9.2.1 L_return（收益端）

```
L_return = 0.35 × (1 - RankIC_normalized)
         + 0.25 × (1 - DirectionAccuracy)
         + 0.20 × CalibrationError
         + 0.15 × ExtremePredictionPenalty
         + 0.05 × ExcessReturnShortfall

其中：

RankIC_normalized:
  对池内N只股票：
    predicted_rank = argsort(scores, descending)
    actual_rank = argsort(actual_returns, descending)
    spearman_r = corr(predicted_rank, actual_rank)
    RankIC_normalized = (spearman_r + 1) / 2  # 映射到[0,1]
  → 测"评分高的股票，未来收益真的更高吗"

DirectionAccuracy:
  predicted_dir = score >= 50 ? +1 : -1
  actual_dir = sign(actual_return - benchmark_return)
  accuracy = mean(predicted_dir == actual_dir)
  → 测"看多/看空的判断，方向对不对"

CalibrationError（Expected Calibration Error）:
  将评分分桶：[0-20), [20-40), [40-60), [60-80), [80-100]
  对每个桶：
    expected = bucket_center_normalized_return
    actual = mean(actual_returns in bucket)
    bucket_error = |actual - expected|
  ECE = Σ(bucket_error × bucket_weight)
  → 测"高分是否真对应高收益"

ExtremePredictionPenalty:
  对score >= 80的预测：if actual_return < 0 → penalty += score/100
  对score <= 20的预测：if actual_return > 0 → penalty += (100-score)/100
  → 测"强烈推荐/强烈看空时，系统是否在误导"

ExcessReturnShortfall:
  excess = strategy_return - benchmark_return
  if excess < 0: penalty = |excess| / max_acceptable_shortfall
  → 测"相对基准的超额收益"
```

#### 9.2.2 L_risk（风险端）

```
L_risk = 0.35 × drawdown_penalty
       + 0.25 × volatility_penalty
       + 0.20 × downside_deviation_penalty
       + 0.20 × consecutive_loss_penalty

drawdown_penalty（最大回撤映射）:
  MDD < 10%:      0
  MDD 10%-20%:    (MDD-0.10)/0.10 × 0.5  → [0, 0.5]
  MDD 20%-40%:    0.5 + (MDD-0.20)/0.20 × 0.5  → [0.5, 1.0]
  MDD > 40%:      1.0

volatility_penalty（年化波动率映射）:
  Vol < 15%:      0
  Vol 15%-30%:    线性映射到[0, 1]
  Vol > 30%:      1.0

downside_deviation_penalty:
  只计算负收益的标准差（下跌波动比上涨波动更值得惩罚）
  同理线性映射

consecutive_loss_penalty:
  连续3次调仓亏损: 0.3
  连续5次: 0.7
  连续8次+: 1.0
```

#### 9.2.3 L_structure（结构端）

```
L_structure = 0.30 × concentration_penalty
            + 0.25 × turnover_penalty
            + 0.25 × cash_drag_penalty
            + 0.20 × sector_diversification_penalty

concentration_penalty:
  HHI = Σ(每只股票权重²)
  penalty = max(0, (HHI - HHI_threshold) / (1 - HHI_threshold))

turnover_penalty（换手率）:
  短线: turnover > 250%/月 → penalty
  中线: turnover > 50%/月 → penalty
  长线: turnover > 15%/月 → penalty

cash_drag_penalty:
  现金比例 > 30%且持续 > 1个月 → penalty
  （熊市中主动减仓导致的现金增加不惩罚，需要区分原因）

sector_diversification_penalty（申万一级行业分组）:
  单一行业 > 40%: 轻度惩罚
  单一行业 > 60%: 中度惩罚
  单一行业 > 80%: 重度惩罚
```

### 9.3 各模块Loss独立计算

```
必须独立产出的模块Loss：
  1. stock_short_term（A股短线）
  2. stock_medium_term（A股中线）
  3. stock_long_term（A股长线）
  4. stock_risk_gate（风险门效果）
  5. consistency（一致性/稳定性）
  6. fidelity（保真度，评测模型 vs 生产模型）
  7. agent_ablation（消融实验结果）
  8. evaluation_model_fidelity（评测配置与生产配置的保真度）
```

### 9.4 风险门Loss

```
gate_effect = {
  "收益变化": gate_on_return - gate_off_return,
  "回撤改善": gate_off_MDD - gate_on_MDD,
  "错杀率": 被gate截断但实际收益 > 0的比例,
  "漏放率": 未被gate截断但实际收益 < -10%的比例
}
```

### 9.5 保真度Loss

```
fidelity_metrics = {
  "action_flip_rate": 快模型与生产模型动作不一致的比例,
  "score_drift": mean(|score_fast - score_production|),
  "topK_overlap": len(intersection(topK_fast, topK_production)) / K,
  "rank_drift": spearman_r(rank_fast, rank_production)
}

如果保真度Loss过大（action_flip_rate > 15% 或 topK_overlap < 70%），
报告中必须明确标注：
  "本次评测配置与生产配置差异过大，结论仅供趋势参考，
   不建议直接据此做系统级优化。"
```

## 10. 统计检验与显著性

### 10.1 核心方法：Bootstrap置信区间

```
对N个anchor date的ΔLoss值:
  1. 有放回抽取N个，计算平均ΔLoss
  2. 重复10000次
  3. 取2.5%分位数和97.5%分位数 → 95%置信区间
  4. 区间完全在0上方 → ★★★（显著正贡献）
  5. 区间包含0      → ☆（不显著）
  6. 区间完全在0下方 → 显著负贡献
```

### 10.2 补充方法：Permutation Test

```
随机交换"全agent"和"消融agent"的标签10000次
得到随机ΔLoss的分布
真实ΔLoss在随机分布的95%分位之外 → 统计显著
```

### 10.3 显著性呈现

| ΔLoss范围 | 95% CI | 显著性 | 星级 | 评价 |
|-----------|--------|--------|------|------|
| > +0.03 | 完全>0 | p < 0.05 | ★★★ | 强正贡献 |
| +0.01 ~ +0.03 | 完全>0 | p < 0.05 | ★★ | 正贡献 |
| -0.01 ~ +0.01 | 包含0 | p > 0.05 | ☆ | 不显著 |
| -0.03 ~ -0.01 | 完全<0 | p < 0.05 | ↓ | 负贡献 |
| < -0.03 | 完全<0 | p < 0.05 | ↓↓↓ | 强负贡献 |

### 10.4 实盘统计的特殊处理

```
短线实盘消融（每天产生观测值）：
  观测值不完全独立（每天重新同步持仓）
  → 使用Cluster Bootstrap（以周为单位Bootstrap，保持周内结构）

中长线实盘：
  不做实盘消融，统计显著性依赖回测
```

## 11. "反向传播"优化机制

### 11.1 核心认知

Agent系统没有梯度，但有"诊断→定位→方案→验证"闭环：

```
Loss → 归因分析 → 定位问题源 → 生成修改方案 → 验证Loss变化
```

### 11.2 问题分类体系

| 类别 | 界定 | 自动程度 | 示例 |
|------|------|---------|------|
| **PARAM_TUNE** | 改配置文件中的数值参数 | 全自动 | scorer权重、risk gate阈值、策略持仓数 |
| **PROMPT_PATCH** | 修改某个Agent的prompt片段 | 半自动（系统生成+人工审批） | 行业指引、打分规则说明 |
| **LOGIC_FIX** | 修改某段代码的业务逻辑 | 半自动（系统生成patch+人工审批） | cache恢复逻辑、signal提取正则 |
| **ARCH_CHANGE** | 涉及Agent架构/数据流 | 全人工（系统只出方案） | 新增Agent、改变依赖关系 |
| **RESEARCH** | 需要深入研究才能确定 | 全人工（系统只出研究报告） | 模型选择、权重体系是否合理 |

### 11.3 PARAM_TUNE（全自动）

**流程**：
  1. Loss引擎检测到某参数相关维度表现差
     例：short_term_scorer的volume_liquidity子维度与收益相关性≈0
  2. 参数空间定义（见下方完整参数空间表）
  3. 贝叶斯优化搜索：在训练期回测上搜索最优权重组合
  4. 验证：候选权重在验证期上测试，必须满足：验证期Loss < 原Loss - 1个标准差
  5. 实施：修改配置文件（不修改业务代码）
  6. 追踪：improvement_log记录，下次检查验证

### 11.7 改动安全网（防越改越差）

```
1. 改动基线：改动前Loss = 最近N次检查的平均Loss
   改动后Loss < 改动前Loss - 1σ → 通过；否则回滚

2. 回滚机制：
   - 所有改动记录在improvement_log
   - 改动前自动保存当前状态（git stash）
   - 改动后Loss恶化 > 阈值 → 自动回滚

3. 冷却期：
   同一模块的自动改动至少间隔N天/N次检查
   防止反复横跳

4. 累积监控：
   连续3次改动后Loss未改善 → 停止该模块自动优化 → 标记为"瓶颈" → 生成人工报告

5. 方向一致性检查：
   如果上次建议"增加fundamental权重"、这次又建议"降低" → 矛盾 → 不自动执行 → 人工判断

6. 候选分支隔离：
   所有自动修改先在candidate分支进行，绝不直接写入stable
```

```python
            "verification_method": "中线回测验证期",
            "risk_level": "low"
        }
    ],
    "narrative_summary": "自由文本，但所有数值必须来自上述结构化证据..."
}
```

### 12.3 代码验证层

LLM输出后，代码自动执行以下验证：

```
1. 数值溯源验证：
   - 扫描LLM输出中的所有数字
   - 确认每个数字都能在输入的指标JSON中找到对应项
   - 对找不到的数字 → 标记为"疑似幻觉" → 用输入数据中的最近似值替换或删除

2. 实体验证：
   - 扫描LLM输出中的Agent名称、文件名、路径
   - 确认每个实体在系统中真实存在
   - 确认文件路径在项目中存在

3. 逻辑一致性验证：
   - 如果claim说"fundamental正贡献"但delta_L为负 → 标记矛盾
   - 如果suggestion说"修改文件X"但X不在白名单 → 降级为MANUAL

4. 比较验证：
   - 如果LLM说"A > B"，代码验证是否真的A > B
```

### 12.4 Self-Consistency（自洽性校验）

```
对关键分析（如agent贡献排序、优化建议类型判断）：
  1. 以不同seed运行2次
  2. 比较两次输出的关键结论是否一致
  3. 不一致的关键结论 → 降级confidence为LOW → 标记"需人工判断"
  4. 仅两次都一致且验证通过的结论标记为HIGH
```

### 12.5 置信度标注

LLM输出的每个结论必须标注置信度：
- **HIGH**：有统计显著性+多维度交叉验证+自洽性通过
- **MEDIUM**：有数据支持但未达到全部验证标准（如样本不足、仅单维度支持）
- **LOW**：推断性结论，缺乏直接数据支持，需人工验证

LOW置信度的结论**不触发**任何自动修改行为。

## 14. 工程优化与缓存架构

### 14.1 采用

| 优化项 | 状态 | 说明 |
|--------|------|------|
| 内存级缓存共享 | ✅ | 进程内dict，启动时加载已有文件缓存，异步刷盘 |
| 分析Agent只跑一次 | ✅ | Line 0完整运行，其余消融线读缓存；跨期限共享 |
| 分级打分频率 | ✅ | 按持仓/候选/观察/边缘分层，不同期限不同频率 |
| 周一全量跑（短线池） | ✅ | 先保留，成本敏感时可通过配置降频或关闭 |
| 缓存命名空间隔离 | ✅ | 生产模型M1/M3缓存与评测模型M5缓存物理隔离；精筛池产物共享给生产，日常模拟盘产物严格隔离 |
| Tushare不可用硬报错 | ✅ | 数据获取时若Tushare不可用且缓存无精确匹配数据，立即报错阻断，禁止使用不完整/不可靠数据产出结论 |

### 14.2 不采用

| 优化项 | 状态 | 原因 |
|--------|------|------|
| 批量合并打分（8合1） | ❌ | 破坏消融线评分独立性，污染实验基础 |
| Prompt压缩 | ❌ | 影响评分质量，不牺牲核心能力 |
| Tushare不可用时降级为估算数据 | ❌ | 不可靠数据导致不可靠结论，宁可不运行也不错运行 |

### 14.3 缓存架构

```
缓存层级：

L1（内存）:
  - 进程内全局dict
  - Key: {agent_name}_{stock_code}_{as_of_date}_eval
  - 所有线共享，跨期限共享
  - 读命中直接返回，无IO

L2（磁盘）:
  - data/eval/cache/{agent}_{code}_{date}_eval.json
  - 异步写入，不阻塞主流程
  - 启动时预加载到L1

缓存Key设计（评测专用）:
  格式: {agent_name}_{stock_code}_{as_of_date}_eval
  示例: fundamental_analysis_sh.603871_2026-06-19_eval.json
  与生产缓存区分（_eval后缀），互不污染
  **缓存命名空间机制**:
    - 生产环境（M1/M3）: 写入 data/intermediate_cache/，文件名无后缀
    - 评测环境（M5）: 写入 data/eval/cache/，文件名追加 _eval 后缀
    - 通过 cache_utils.set_cache_namespace() 切换，线程安全
    - 精筛池筛选使用生产模型 M1/M3，命名空间=None，
      产物写入生产缓存，可被主投顾系统直接复用，避免重复LLM调用
    - 日常模拟盘/回测使用评测模型 M5，命名空间="eval"，
      产物写入评测缓存，与生产缓存完全物理隔离，
      绝不允许M5产物污染生产环境
    - finish_batch() 自动恢复命名空间为None，防止泄漏

  四场景共享/隔离判定表：
    ┌──────────────────────────┬──────────┬──────────────────────────────┐
    │ 场景                     │ 是否复用  │ 原因                         │
    ├──────────────────────────┼──────────┼──────────────────────────────┤
    │ 生产缓存 → 精筛池        │ ✅ 可复用 │ 同模型M1/M3，同目录，同key    │
    │ 精筛池产物 → 生产环境    │ ✅ 可复用 │ 精筛池agent通过cache_utils    │
    │                          │          │ 自动写入intermediate_cache/   │
    │ 模拟盘/回测产物 → 生产   │ ❌ 隔离   │ M5模型写入data/eval/cache/    │
    │                          │          │ + _eval后缀，生产查不到       │
    │ 生产缓存 → 模拟盘/回测   │ ❌ 隔离   │ 模拟盘查_eval后缀文件，       │
    │                          │          │ 生产缓存无此后缀；即使日期     │
    │                          │          │ 代码相同，模型不同不可混用     │
    └──────────────────────────┴──────────┴──────────────────────────────┘

跨期限共享：
  同一只股票、同一天、同一个agent → 只跑一次
  不论短线/中线/长线，不论哪条消融线
  示例：601888（中国中免）同时在短中长三个池中
    → fundamental_analysis 只跑1次，三个期限共享结果

独立Agent缓存分解：
  每次完整管线运行后，结果被分解为7个agent的独立缓存条目，
  各自使用对应的TTL：
    - fundamental_analysis: 15天 TTL — 财报数据低频变化，15天内同一股票不重跑
    - value_analysis: 7天 TTL — 估值框架稳定，7天内不重跑
    - quality_risk_analysis: 7天 TTL — 质量指标低频，7天内不重跑
    - technical_analysis: 1天 TTL — 日线价格变化，每天刷新
    - news_analysis: 1天 TTL — 新闻时效性强，每天刷新
    - event_analysis: 1天 TTL — 事件驱动，每天刷新
    - moneyflow_analysis: 1天 TTL — 资金流日频数据，每天刷新
  此外，完整管线结果同时写入 full_analysis 缓存（1天TTL）作为快速回退。

缓存全命中跳过管线：
  当某只股票的7个agent缓存全部在各自TTL内且未过期时，
  后续请求直接使用缓存数据，不再调用LLM管线。
  实际效果：对于稳定评分的股票，fundamental_analysis每15天只跑1次，
  value/quality_risk每7天只跑1次，仅技术面/新闻/事件/资金流每日刷新。

缓存TTL（沿用生产系统TTL + 分级频率覆盖）：
  fundamental_analysis: 15天 | value_analysis: 7天
  quality_risk_analysis: 7天 | technical_analysis: 1天
  news_analysis: 1天 | event_analysis: 1天
  moneyflow_analysis: 1天 | full_analysis: 1天
  注：分级打分频率可能提前触发刷新（覆盖TTL）
```

### 14.4 分级打分频率

```
短线精筛池（100只）：
  持仓股（~10只）:     每天跑 — 每日必调仓
  高波动候选（~15只）:  每天跑 — score 60-80，随时可能买入
  稳定高分区（~20只）:  每3天跑 — score > 75且近10日波动 < 5%
  中位区（~30只）:      每5天跑 — score 45-65
  稳定低分区（~25只）:  每7天跑 — score < 45且无明显改善
  周一全量：每周一全量跑一次

中线精筛池（80只）：
  持仓股（~8只）:       调仓日必跑（每周五）
  候选股（~15只）:      每周跑 — score > 60
  观察股（~25只）:      每2周跑 — score 45-60
  边缘股（~32只）:      每月跑 — score < 45
  调仓日全量：每周五全量跑

长线精筛池（60只）：
  持仓股（~5只）:       调仓日必跑（每月末）
  候选股（~10只）:      每月跑
  观察股（~20只）:      每季跑
  调仓日全量：每月末全量跑

升频触发（纯代码监控器）：
  - 某股票连续2次评分变化 > 10分 → 提升一个频率等级
  - 某股票近5日涨跌幅 > 15% → 提升一个频率等级
  - 持仓股score < 50 → 标记，下次必须跑
  - 某股票连续3次评分变化 < 3分 → 降低一个频率等级

调度实现：
  - 使用年内日序号（day_of_year）取模判定当天各等级是否执行：
    "3day"等级 → day_of_year % 3 == 0 时执行
    "5day"等级 → day_of_year % 5 == 0 时执行
    "7day"等级 → day_of_year % 7 == 0 时执行
  - 持仓股始终不受分级限制，每个调仓日必分析
  - 周一全量覆盖所有等级，消化周末消息面变化
  - 非分析日股票使用池内上次缓存评分，不调用Agent

注意：
  - 中线和长线的LLM自由线不经过任何Agent分析中间产物，
    LLM直接调用MCP工具获取数据后自主决策
```

#### 16.1.9 进程进度条规范

所有耗时操作（预计执行时间 > 1秒）必须在Streamlit前端展示用户可见的进度条，具体要求：

```
进度条适用条件：
  - 耗时 > 1秒（预估或实测）：必须加进度条
  - 耗时 ≤ 1秒的瞬发操作：不加进度条，避免UI闪烁

必须展示进度条的操作清单：
  1. 一键完整检查 — 显示阶段进度：
     "检测缺失日 → 补回历史(N/M天) → 结算历史 → 各期限调仓 → 收盘结算 → 生成报告"
  2. 精筛池更新 — 分4层显示：
     "Layer 0 硬筛... → Layer 1 批量粗筛(N/M批)... → Layer 2 快筛... → Layer 3 精筛..."
  3. 回测运行 — 显示anchor进度：
     "回测中: N/M 个时点 (XX%) [训练期/验证期/测试期]"
  4. 优化分析 — 显示分析进度：
     "分析中: 计算Loss → 归因分析 → LLM诊断 → 生成建议"

进度条实现规范：
  - 使用 Streamlit 的 st.progress() + st.status() 组件
  - 进度值用 0.0~1.0 浮点数，精确到 1%
  - 每个阶段完成后即时更新进度条，不得等到全部完成才更新
  - 支持取消按钮（st.button("取消")），点击后优雅终止流程
  - 若某阶段失败，进度条停在失败阶段，显示红色错误信息，不继续后续阶段
  - 进度文字包含"已完成/总计"和"预计剩余时间"（基于已完成阶段的平均速度估算）
```

## 20. 工程红线

### 20.1 必须遵守的CLAUDE.md约束

1. 不新增AkShare依赖 — 数据源以Tushare为主
2. 不硬编码模型名 — 必须通过`model_config.py`读取配置
3. 不破坏现有`signal_pack`架构 — 新增`DecisionPack`与现有体系并行
4. 不跳过`tool_cache` — 所有复用MCP工具的调用继续走缓存
5. 不移除`merge_dicts` reducer — 并行Agent状态写入保持兼容
6. 不假设LLM输出类型稳定 — 所有新增结构做normalize/fallback
7. 兼容Windows/WSL — 不引入仅Linux可用的进程/文件锁实现

### 20.2 评测系统自身红线

1. 任何用户可见的最终推荐结果必须有`decision_pack`
2. 历史回放必须显式带`as_of_date`
3. 回测新闻/事件数据不可完整回放时必须标记`pit_mode`
4. **所有Loss必须由代码计算，禁止LLM生成数值**
5. 自动优化不直接改stable主分支
6. 自动优化必须先测试，测试失败必须回滚
7. 如果无法确认某个细节：优先配置化 > 增加TODO/fallback > 不强行假设
8. **DeepSeek V4 Pro的所有输出必须经过反幻觉验证层**（第12章）
9. 评分配置与生产配置的保真度指标必须持续监控，偏差过大时报告中明确标注
10. **Tushare不可用时硬报错**：任何数据获取操作，若Tushare不可用（网络不通/限流/服务端错误）且缓存中无精确匹配该查询的数据，必须抛出明确错误并阻断当前流程，禁止回退到模拟值/估算值/默认值/随机值。宁可系统报错停止，也不可用不可靠数据得出不可靠结论。
